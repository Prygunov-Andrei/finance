## 0. Преамбула

Работа со сметами в ERP Август сейчас не даёт стабильного автоматического качества: подбор работ и материалов в проде не запускался ни на одной из 4 имеющихся смет, база знаний (ProductKnowledge) заполнена на 582 записи со статусом PENDING, ни одной VERIFIED; ProductWorkMapping пустой. Модуль смет тесно завязан на ERP по справочникам, авторизации, файлам проекта, кадровым ролям — любое развитие тянет за собой половину ERP.

Решение — выделить смёточный функционал в обособленный продукт **ISMeta** («айсмета») с:
- собственной БД, фронтендом, LLM-инструментами;
- API-границей с ERP (REST + webhooks, без прямого доступа к чужой БД);
- перспективой использования в публичной части сайта Августа и, позднее, в виде коробочного продукта для смежных компаний.

Первым и основным потребителем ISMeta будет ERP Август. Публичный режим и коробка — параллельное проектирование и последующие итерации.

Документ фиксирует принятые архитектурные решения после трёх раундов согласования и раунда критической правки. Он не содержит оценок в человеко-днях и не назначает сроки — это работа следующего уровня планирования.

## 1. Видение и принципы

### 1.1 Что такое ISMeta

ISMeta — самостоятельный продукт для подготовки коммерческой сметы. Получает на вход структурированные данные спецификации (от сервиса распознавания или загрузкой из Excel), выполняет подбор работ и материалов, даёт интерфейс редактирования и экспорта, отдаёт готовую смету в контурную систему (ERP Август или аналог у клиента) для оформления договора.

ISMeta не занимается: распознаванием документов, контрактами и актами, бухгалтерией, управлением проектной документацией, монтажными нарядами. Всё это остаётся на стороне ERP или соседних сервисов.

### 1.2 Целевая аудитория в порядке приоритета

1. **Сметчик ERP Август** — основной пользователь в MVP. Работает с реальными сметами нашей компании, помогает обкатать продукт.
2. **Посетитель публичного сайта Августа** — получает быстрый предварительный расчёт по загруженной спецификации. Цель — лидогенерация и демонстрация возможностей, не замена сметчика.
3. **Сметчик коробочного клиента в смежной отрасли (ОВиК/СС)** — появляется не раньше этапа 3. До этого концепция учитывает коробочный сценарий архитектурно, но не реализует.

Другие отрасли (строймонтаж, электромонтаж, сантехника и прочее) — вне скоупа концепции. Не закладываем, не отрицаем — отдельный разговор.

### 1.3 Что даёт выделение в отдельный продукт

Конкретные инженерные результаты, без оценок в процентах:
- возможность разрабатывать и тестировать сметный функционал, не перекатывая весь ERP;
- переносимость в публичную часть сайта и в чужие инсталляции;
- собственная база знаний подбора, собственный LLM-агент, собственные политики обновления;
- архитектурная подготовка к multi-tenant SaaS на будущее.

Подтверждённых цифр по ускорению сметчика, сокращению штата и т.п. у нас нет. Собираем их по ходу эксплуатации MVP — это задача метрик (раздел 8).

### 1.4 Принципы разработки

1. **Качество выше скорости.** Не «как-нибудь, потом перепишем». Каждое решение проходит через концепцию и план.
2. **Документация — сразу, для всех ролей** (разработчик, сметчик, админ-внедренец). Писать по ходу, не отложить на «когда продадим».
3. **Нет обратной совместимости со старым модулем смет в ERP.** ERP сейчас не в продуктивной эксплуатации, никто с ним не работает, 4 существующие сметы остаются как архив.
4. **Тонкая граница ответственности.** ISMeta не трогает договоры, акты, проекты, бухгалтерию. Каждая сущность принадлежит ровно одному сервису.
5. **Multi-tenancy с первого дня** — через UUID-идентификатор `workspace_id` на всех таблицах. Плата сейчас — мелкая; плата потом — переписать всё.
6. **Параллельная подготовка соседей или моки.** ISMeta зависит от 6 соседних сервисов (перечислены в разделе 6). Каждый либо разрабатывается параллельно с ISMeta, либо мокается в тестовой среде до готовности реального.
7. **Готовность коробки — это цель этапа 3+, не MVP.** В MVP ISMeta тесно интегрирована с ERP Август, и мы это принимаем.

## 2. Границы продукта

### 2.1 Что входит в ISMeta

- Иерархия: `Workspace` → `Folder` → `Estimate` → `Section` → опциональная `Subsection` → `Item`.
- Линейное версионирование смет (`parent_version`, `version_number`), на каждой строке `source_item_id` для сравнения и переноса данных между версиями.
- Импорт сметы: от сервиса распознавания (по API), из Excel/CSV.
- Подбор работ (8-уровневый pipeline перенесён из текущего ERP).
- Подбор материалов (цены из ERP-каталога и истории закупок, запрашивается live).
- База знаний: `ProductKnowledge`, `ProductWorkMapping`, `.md`-правила. Живёт внутри ISMeta.
- LLM-агент с одним инструментом в MVP — «Переподбор аналога по ТТХ». Остальные инструменты — этапы 2 и 3.
- Excel-цикл: экспорт со скрытым `row_id` и hash-полем для обнаружения разрушения идентификаторов → правка в Excel → импорт с diff-предпросмотром.
- Публичный режим: отдельный URL, OTP-логин, интерактивный виджет настройки ставки часа, материалы в РРЦ.
- Аудит-лог изменений сметы — 1 год хранения (сокращено с 3 после критики: для миллиона записей на сметчика нужен либо партиционинг, либо короткий срок).
- Дашборд метрик (упрощённый, без materialized views в MVP).
- Интеграция с ERP по REST + webhooks (outbox pattern в ERP, idempotent приёмник в ISMeta, polling-fallback каждые 60 секунд).

### 2.2 Что НЕ входит в ISMeta

- **Распознавание спецификаций.** Отдельный сервис (раздел 6, приложение В.1).
- **Проектная документация** (`Project`, `ProjectFile`, `ProjectNote`, `ProjectFileType`). Не нужна ISMeta.
- **ТКП**, **монтажные сметы**, **договоры**, **акты КС-2**, **допсоглашения**, **накопительные сметы**, **закупки**, **бухгалтерия**. Всё в ERP.
- **Каталог товаров и история цен** — источник правды в ERP, ISMeta использует кеш Product (минимальные поля) и live-запросы за деталями и историей.
- **Прайс работ** — в ERP, ISMeta дёргает live при подборе (238 строк в проде, кеш не нужен).
- **Нормативная база (СНиП/ГОСТ/ГЭСН/ТСН/МДС)** — отдельный сервис RAG, в MVP отсутствует.
- **Электронная подпись** — не в MVP, см. приложение Г.
- **Параметризуемые печатные формы** (логотипы, реквизиты клиентов) — упрощённый PDF в MVP, полная параметризация — приложение Г.
- **Импорт из Гранд-Сметы, Турбосметчика, АБК-5** — приложение Г.

### 2.3 Цикл «ISMeta → ERP»: границы ответственности

ISMeta не создаёт в ERP ни Contract, ни ContractEstimate, ни ContractAmendment самостоятельно. Она только отдаёт снимок сметы как «готовая версия N для договора №X» или как «независимый новый снимок». Что именно оформляется в ERP как ДОП, как новый договор, как обновление существующего ContractEstimate — решает пользователь в ERP (сметчик, юрист, руководитель проекта).

```
ISMeta                                       ERP Август
──────                                       ──────────
[смета, версии v1..vN]
      │
      │ кнопка «Отдать в ERP»
      │ POST /api/v1/ismeta-snapshots/
      │ { estimate_id, version_id,
      │   contract_hint: null | contract_id }
      ▼
                                      ┌─── получен snapshot ───┐
                                      │ ERP-сметчик/юрист      │
                                      │ видит уведомление:     │
                                      │ «ISMeta v2 готова»     │
                                      │ и сам принимает        │
                                      │ решение:               │
                                      │  - новый Contract      │
                                      │  - новая версия        │
                                      │    ContractEstimate    │
                                      │    как ДОП             │
                                      │  - обновление текущей  │
                                      │    ContractEstimate    │
                                      │                        │
                                      │ Реквизиты ДОП (номер,  │
                                      │ дата, основание)       │
                                      │ заполняются в ERP.     │
                                      └────────────────────────┘
                                                  │
                                                  │ webhook обратно:
                                                  │ contract.signed
                                                  │ { version_id, contract_id }
                                                  ▼
[в ISMeta у версии v2 появляется
 пометка «передана в ERP,
 договор №123 от 2026-05-01»]
```

Точечные правки, которые ERP-пользователь делает после подписания договора (CONSUMABLE, ADDITIONAL, EstimatePurchaseLink), в ISMeta **не возвращаются вообще никогда**. Это сознательное упрощение.

Сметчик в ISMeta у себя видит список версий. При создании новой версии v_n+1 копия строится с предыдущей (с `source_item_id`). Если ERP-сметчик хочет учесть ранее сделанные точечные правки — он видит их список в ERP при создании ContractAmendment и галочкой переносит нужные в новый ContractEstimate. В ISMeta они не переносятся (их там никогда и не было).

## 3. Карта данных и движение информации

> Подробно: [`specs/01-data-model.md`](./specs/01-data-model.md), [`specs/02-api-contracts.md`](./specs/02-api-contracts.md), [`specs/03-webhook-events.md`](./specs/03-webhook-events.md).

### 3.1 Что хранит ISMeta (в собственной БД)

| Группа | Таблицы |
|---|---|
| Тенант | `Workspace` (id — UUID), `WorkspaceMember` |
| Иерархия | `Folder` (опциональный `external_ref` на Object из ERP, синхронизируется webhook'ом) |
| Смета | `Estimate`, `EstimateSection`, `EstimateSubsection` (опциональная в UI), `EstimateItem`, `EstimateCharacteristic`, `EstimateMarkupDefaults`, `ColumnConfigTemplate` |
| Версии | поля `parent_version`, `version_number`, `source_item_id` на строке |
| База знаний | `ProductKnowledge`, `ProductWorkMapping`, файлы `data/knowledge/products/*.md` |
| Кеш справочников | `ProductCache` (минимальные поля, инвалидация webhook + TTL 15 минут) |
| LLM-агент | `ChatSession`, `ChatMessage`, `AgentContext` (по смете, а не по сессии — см. 4.4) |
| Аудит | `AuditLog` (1 год, без партиционинга) |
| Импорт | `ImportSession` с версионированием `row_id` |
| Outbox/Inbox | `processed_events` (TTL 14 дней) для idempotency |

Все таблицы содержат `workspace_id: UUID`. FilterBackend в DRF блокирует доступ по чужим workspace на уровне ViewSet.

### 3.2 Что ISMeta не хранит, дёргает по API

| Сущность | Где | Как использует ISMeta |
|---|---|---|
| Object | ERP | `external_ref` на Folder, обогащается при отображении |
| Counterparty, LegalEntity | ERP | live-запрос для шапки сметы и экспорта |
| Project, ProjectFile | ERP | не используется |
| Contract, ContractEstimate, Act | ERP | только снимок наружу, обратно — webhook `contract.signed` |
| Invoice, ProductPriceHistory | ERP | live при подборе материалов |
| Product детали | ERP | по конкретным id, кроме кешированных полей |
| WorkItem (прайс) | ERP | live при подборе (238 строк — кеш не нужен) |
| Нормативная база | RAG-сервис | live, опционально |

### 3.3 События (webhook от ERP к ISMeta)

Outbox pattern на стороне ERP, idempotent receiver на стороне ISMeta по `event_id`, polling fallback каждые 60 секунд (не 5 минут, как было в первой версии — скорректировано после критики о UX-задержках).

| Событие | Эффект в ISMeta |
|---|---|
| `product.updated` | инвалидация ProductCache по id |
| `product.archived` | пометка в кеше, UI-предупреждение в активных сметах, использующих этот product |
| `pricelist.activated` | пометка «нужно перепересчитать» на сметах, ссылающихся на предыдущий PriceList |
| `worker_grade.rate_changed` | уведомление сметчику по каждой смете, использующей этот грейд (критичное поле, добавлено после критики D2) |
| `contract.signed` | пометка «передана в ERP, договор №…» на соответствующей версии сметы |
| `contract.terminated` | пометка снимается (возврат к торгам) |
| `object.created/updated/deleted` | синхронизация дерева Folder |
| `counterparty.updated` | инвалидация (если кеш будет добавлен; в MVP кеша контрагентов нет) |

### 3.4 Поведение при недоступности ERP

- CRUD по сметам, версиям, ProductKnowledge, LLM-чату — работает (собственная БД ISMeta).
- Кеш Product используется как есть, с баннером «данные каталога от ЧЧ:ММ». TTL 15 минут — компромисс между свежестью и доступностью.
- Live-запросы (WorkItem, ProductPriceHistory, Counterparty, LegalEntity) — возвращают понятную ошибку «ERP недоступен». Сметчик может либо отложить операцию, либо использовать ранее подгруженные данные (если они были), либо ввести значения вручную (см. ниже).
- Валютные курсы ЦБР (USD/EUR/CNY): в норме берутся из ERP. При недоступности ERP сметчик вводит курс вручную в шапке сметы.
- Отдача сметы в ERP: запрос ставится в очередь retry с экспоненциальным backoff, сметчик видит статус «ожидает связи с ERP».
- Жёсткий offline-режим (сутки работы без ERP) — не делаем.

### 3.5 Контракты API (что должно появиться как артефакты)

Все контракты оформляются как OpenAPI 3.1 спецификации в репозитории, версионируются по semver (`v1`, `v2`, deprecated-заголовки).

| Контракт | Владелец | Содержание |
|---|---|---|
| **ISMeta → сервис распознавания** | ISMeta | POST файл → `{ items[], document_meta }` без FK на ERP |
| **ERP → ISMeta (snapshot)** | ERP | POST снимок сметы с разделами, строками, ценами, прибыльностью, человеко-часами, обоснованиями подбора |
| **ISMeta → ERP (catalog/pricelist API)** | ERP | GET Product, GET WorkItem, GET ProductPriceHistory, с пагинацией и фильтрами |
| **ERP → ISMeta (webhooks)** | ERP | POST `/ismeta/webhooks/` с `event_id`, `event_type`, payload |
| **ISMeta виджет ↔ ERP (auth)** | ERP | service-to-service `POST /erp-api/ismeta/issue-jwt` → короткий JWT для юзера + refresh-токен |

### 3.6 Авторизация между сервисами (уточнение после критики D5)

- При настройке интеграции ERP выдаёт ISMeta `master_token` (долгоживущий, хранится в секретах).
- Когда сметчик в ERP-UI открывает виджет ISMeta, фронт ERP вызывает `POST /erp-api/ismeta/issue-jwt` — ERP создаёт пару (access JWT 15 минут, refresh 8 часов) и отдаёт фронту.
- Виджет работает с access JWT, при 401 автоматически обновляется через refresh.
- Сессия в 8 часов покрывает рабочий день без повторных логинов.
- Auth для webhook'ов ERP→ISMeta: HMAC-подпись в заголовке `X-Webhook-Signature` на shared secret, IP allow-list на стороне ISMeta (добавлено после критики F10).

### 4.3 Кеш и консистентность

**Product cache.** Минимальные поля: `id, name, normalized_name, unit, category_id, default_price, status`. Инвалидация по webhook `product.updated` / `product.archived`. TTL 15 минут (не 24 часа) — компромисс после критики D1: при потере webhook'а максимум 15 минут рассинхронизации. UI явно показывает «последнее обновление кеша: ЧЧ:ММ».

**WorkItem cache не делаем.** 238 строк в проде, live-запрос мгновенный. Если коробочный клиент принесёт 10 000+ — добавим кеш.

**ProductPriceHistory cache не делаем.** Данные быстро устаревают, дёргаем live по конкретным product_id.

Дополнительно заложена возможность логической репликации (CDC) для Product и WorkItem — на случай, если webhook+TTL окажутся недостаточны в продакшене. Но в MVP — только webhook+TTL.

### 4.4 LLM-агент

**Архитектура:** ReAct-style с tool use (LLM вызывает инструменты, получает результат, рассуждает, продолжает).

**Инструменты в MVP — два:**
- `get_item(item_id)` — полная информация по строке сметы (название, ТТХ, подобранный work_item, цена, наценка).
- `find_alternatives_by_specs(item_id)` — top-N аналогов по ТТХ из каталога ERP.

Никаких `propose_change` / `apply_change` в MVP (после критики E4). Агент только предлагает варианты в чат, сметчик вручную кликает «заменить» в UI сметы — эта кнопка уже есть в CRUD-API.

**Остальные инструменты** (`get_estimate_summary`, `get_section`, `search_knowledge`, `get_supplier_prices`, пропозал-коммиты) — этап 2.

**Память агента:** на уровне сметы, а не сессии (после критики E5). Хранится последние 20 сообщений чата по данной смете, независимо от того, когда сметчик к ней вернулся. Долгосрочная память «сметчик не любит Vaillant» — приложение Г.

**LLM-провайдер:** настраиваемый на уровне Workspace. По умолчанию — OpenAI gpt-4o (как в текущем ERP). Альтернативы: Gemini, Grok, GigaChat, Yandex GPT, локальный Ollama. Какую модель выбирать — решает клиент при установке. Качество подбора на локальных LLM ожидается ниже, но без бенчмарков на golden set конкретные цифры не называем (после критики G3).

**Экономика токенов** (добавлено после критики F1):
- Каждый LLM-вызов логируется в `llm_usage` с `tokens_in, tokens_out, cost_usd, workspace_id, task_type`.
- Дашборд расходов в админке ISMeta: стоимость на смету, на сметчика, на workspace, помесячно.
- Целевой ориентир MVP: не более 50 ₽ LLM-затрат на одну смету среднего размера (2 000 строк). Превышение — сигнал на оптимизацию промптов.

### 4.5 Multi-tenancy

Каждая запись имеет `workspace_id: UUID` (не int). UUID с первого дня — это защита от коллизии при будущем слиянии коробочных инстансов в SaaS (объяснение критики B5):

- В коробке у клиента один Workspace. Его `workspace_id` — случайный UUID.
- При переходе в SaaS два клиента с разными UUID не конфликтуют в одной БД — просто суммируются.
- Если бы `workspace_id` был `int = 1`, при слиянии пришлось бы перенумеровывать все FK в десятках таблиц, риск потерять данные.

**Изоляция на уровне приложения:**
- DRF FilterBackend на каждом ViewSet.
- Celery — отдельные очереди или приоритизация per workspace (после критики D7).
- Redis ключи префиксуются `ws:{workspace_id}:...`.
- S3/файловое хранилище: bucket-префикс `ws/{workspace_id}/...`.

**Early-test mode** (после критики G4): уже на этапе 1 поднимаем в dev-окружении ДВА Workspace с разными данными и гоняем все автотесты с обоими. Любая утечка данных между ними — баг этапа 1.

### 4.6 Надёжность webhook'ов

- Outbox pattern в ERP: события записываются в таблицу `outbox`, фоновый воркер отправляет в ISMeta с retry.
- Idempotent receiver в ISMeta: дедупликация по `event_id`, таблица `processed_events` с TTL 14 дней.
- Polling fallback: ISMeta каждые 60 секунд опрашивает ERP `GET /api/sync/events?since=last_event_id` для восстановления при потере webhook'а (изменено с 5 минут после критики D6).
- Мониторинг: алерт если в `processed_events` больше 5 минут не приходит ни одного события при активной работе сметчика.

### 4.7 Optimistic locking строк сметы

Даже при правиле «один сметчик на одну смету» (организационное) технически сметчик может открыть смету в двух вкладках. Чтобы избежать потери изменений:

- Каждая строка `EstimateItem` имеет поле `version: int`, инкрементируется при любом UPDATE.
- При PATCH строки клиент передаёт `If-Match: version`; если не совпадает — 409 Conflict с текущим состоянием строки.
- UI показывает диалог «строка изменилась в другом месте, перезагрузить или перезаписать?».

Полноценный real-time collaboration (CRDT, WebSocket) — вне скоупа.

### 4.8 Аудит-лог

Каждое изменение строки, раздела, наценки, статуса → запись в `AuditLog`: `workspace_id, estimate_id, entity_type, entity_id, user_id, action, old_value, new_value, source` (UI / API / агент / импорт / webhook). Срок хранения — 1 год (сокращено с 3 после критики B7). Просмотр — отдельная вкладка «История» в UI сметы. Без партиционинга и TimescaleDB в MVP (после критики H5 — масштабы не оправдывают).

### 4.9 Безопасность

- **Загружаемые файлы** (PDF/Excel): антивирусное сканирование перед распаковкой (ClamAV или аналог, критика D10). Отдельная sandbox-директория, лимит на размер.
- **Webhook подписи** (критика F10): HMAC `X-Webhook-Signature`, IP allow-list, отклонение запросов без подписи.
- **Секреты**: в env, никаких в БД. LLM API-ключи шифруются at-rest, если хранятся в Workspace-конфиге.
- **Бэкап БД** (критика D9): ежедневный полный дамп + wal-g для incremental. Хранение в S3 с версионированием, 30 дней. Восстановление отработано и задокументировано в Admin Guide.
- **Rollback релизов** (критика F5): все миграции Django обратимы (reverse_code для RunPython). Релизная процедура включает rollback-план.

## 5. Workflow сметчика и публичный режим

> Подробно: [`specs/05-excel-schema.md`](./specs/05-excel-schema.md) (Excel-цикл), [`specs/10-public-mode.md`](./specs/10-public-mode.md) (публичный режим).

### 5.1 Основной сценарий сметчика (MVP)

```
 1. Сметчик заходит в ISMeta через виджет в ERP или standalone-URL.
 2. Создаёт смету: выбирает Folder (синхронизированный с Object из ERP),
    задаёт название, валюту (с ручным курсом, если ERP недоступен),
    юр. лицо (live из ERP), прайс работ, наценки по умолчанию.
 3. Загружает файл спецификации.
    a) Если есть сервис распознавания — он возвращает items[] по контракту.
    b) Если его нет ещё — сметчик загружает Excel.
 4. ISMeta создаёт строки (без цен, без привязок).
 5. Нажимает «Подобрать работы»:
    - Pass 1 (Tier 0-5, синхронно, 40-60 с для 4000 строк — с прогрессбаром);
    - Pass 2 (Tier 6-7, асинхронно через Celery и LLM, минуты, с уведомлением).
 6. Нажимает «Подобрать материалы» — preview по источникам (SupplierProduct,
    ProductPriceHistory), сметчик одобряет построчно или bulk'ом.
 7. Ревьюит результаты:
    - принять авто-подбор;
    - отказаться и переподобрать вручную через search UI;
    - открыть чат с LLM-агентом: «найди аналог по ТТХ».
 8. Корректирует наценки: глобальные → секционные → построчные
    (каскад, как в текущем ERP).
 9. При необходимости — «Открыть в Excel» для объёмных ручных правок:
    экспорт с row_id и hash, редактирование, импорт с diff-предпросмотром.
10. Создаёт версию для клиента, экспортирует PDF.
11. После согласования — кнопка «Отдать в ERP».
    ISMeta пушит snapshot; ERP-пользователь оформляет ContractEstimate/ДОП
    внутри ERP самостоятельно.
12. Webhook `contract.signed` приходит обратно → версия в ISMeta
    становится read-only с пометкой «договор №… от …».
13. Если клиент попросил перерасчёт — кнопка «Новая версия».
    Копирование с source_item_id. Цикл повторяется.
```

### 5.2 Excel-цикл

При экспорте .xlsx содержит скрытые поля:
- `row_id` — стабильный UUID;
- `row_hash` — хэш значимых полей (name, quantity, unit, prices, markup) на момент экспорта.

При импорте:
- Сверка по `row_id`. Если файл потерял скрытые столбцы (critique D3) — диалог «идентификаторы строк потеряны, режим fallback: сверка по имени+модели+количеству, точность ниже».
- `row_hash` сопоставляется с текущим состоянием в БД. Если отличается — значит строка менялась в UI после экспорта, показываем трёхстороннее сравнение (было / правка в Excel / текущее в БД).
- UI показывает diff: столбцы «было», «стало в Excel», «сейчас в БД», галочки применить/отменить по каждой строке.

Google Sheets — приложение Г.

### 5.3 Публичный режим

Отдельный URL (например `https://smeta.august.ru/`), встраивается на публичный сайт или работает как отдельный лендинг.

Поток:
1. Посетитель вводит email → OTP 10 минут.
2. Загружает PDF/Excel спецификации.
3. ISMeta: распознавание → создание сметы → автоподбор. Время — **зависит от объёма PDF и текущей очереди LLM**, показываем прогресс, не обещаем «1-3 минуты» (после критики D4).
4. По готовности посетитель видит:
   - разделы и строки;
   - материалы: РРЦ (через `PublicPricingConfig`, без закупочных цен);
   - работы: человеко-часы по грейдам + виджет «Стоимость часа по грейду» с полями ввода по умолчанию (значения с наценкой). При изменении — мгновенный JS-пересчёт итогов всех работ.
5. Кнопки «Скачать PDF», «Заказать звонок сметчика».
6. Заявка посетителя (в том числе на звонок) — push-событие в ERP, там сметчик работает с ней дальше уже в основном режиме ISMeta.

### 5.4 Коробочный сценарий (для записи, не MVP)

В дистрибутиве:
- `docker-compose.yml` с backend, frontend, postgres, redis;
- `.env.example`;
- минимальный `install.sh` с заполнением ключевых параметров;
- базовый снимок ProductKnowledge и `.md`-правил из нашего ERP.

Внедрение каждого коробочного клиента — индивидуально, с участием нашей команды. Не делаем Kubernetes-манифесты, Helm-чарты, Ansible-роли.

Критический момент: в MVP ISMeta технически работает только в паре с ERP Август (нужны caталог/прайсы/контрагенты). Для полноценной коробки у стороннего клиента нужен упрощённый каталог внутри ISMeta либо готовый совместимый источник у клиента. Это приложение Г и этап 3+ (уточнение после критики G2).

## 6. Окружение ISMeta

> Подробно: приложение В этого документа (список планов соседних сервисов), [`specs/02-api-contracts.md`](./specs/02-api-contracts.md) (API сервиса распознавания и каталога ERP).

ISMeta опирается на 6 соседних сервисов. Каждый либо реализуется параллельно, либо мокается в тестовом окружении.

### 6.1 Сервис распознавания документов (рабочее имя DocAI)

- **Цель:** PDF/Excel/изображение → JSON с items[].
- **Текущее состояние:** частично есть (`backend/llm_services/services/specification_parser.py`, используется в `api_public/tasks.process_public_estimate_request()`).
- **Что нужно:** выделить в отдельное Django-приложение `recognition/` внутри ERP. API без FK на ERP.
- **Приоритет:** критический для MVP.
- **Мок на время разработки:** принимает запрос, возвращает фиксированный JSON (`mock_recognition.json`).

### 6.2 API каталога и прайсов в ERP

- **Цель:** дать ISMeta публичный REST-интерфейс к Product, WorkItem, ProductPriceHistory, WorkerGrade.
- **Текущее состояние:** в ERP есть модели и внутренние views, но публичного API с пагинацией/фильтрами/webhook'ами нет.
- **Что нужно:** OpenAPI-спецификация, эндпоинты, outbox-таблица с событиями изменений.
- **Приоритет:** критический для MVP.
- **Мок:** статические фикстуры на 100 товаров и 30 работ в dev-окружении ISMeta.

### 6.3 LLM-шлюз в ERP

- **Цель:** единая точка конфигурации LLM-провайдеров, учёт токенов и стоимости, квоты на workspace.
- **Текущее состояние:** есть `LLMProvider`, `LLMTaskConfig`, биллинга нет.
- **Что нужно на MVP:** минимально — учёт токенов на запрос (2-3 дня работы), простая таблица `llm_usage`. Квоты и дашборд — итерация 2. Старт — очень упрощённо, доработки по мере нужды (после критики C2).
- **Приоритет:** параллельный с MVP.
- **Мок:** HTTP-клиент ISMeta умеет работать в режиме «замороженный ответ» по фикстурам.

### 6.4 Интеграции с поставщиками

- **Цель:** каталоги поставщиков, RFQ (запросы цен), приём ответов.
- **Текущее состояние:** 1 интеграция (Breez), RFQ-модель, SupplierProduct — 3 055 записей.
- **Что нужно:** расширение только во второй итерации; для MVP ISMeta достаточно текущего состояния через API каталога.
- **Приоритет:** после MVP.

### 6.5 Почтовый парсер

- **Цель:** приём и классификация писем (ответы поставщиков на RFQ, спам/не спам).
- **Текущее состояние:** отсутствует полностью.
- **Приоритет:** после RFQ (этап 2-3).

### 6.6 RAG-база нормативных документов

- **Цель:** СНиП/ГОСТ/ГЭСН для LLM-агента и переподбора по нормам.
- **Текущее состояние:** отсутствует.
- **Приоритет:** не для MVP. Этап 2+ или параллельно, если появится ресурс.
- **В MVP:** LLM-агент отвечает из общего знания провайдера (OpenAI и пр.). Гипотеза, что этого достаточно для переподбора аналогов по ТТХ — проверяется в эксплуатации MVP.

### 6.7 Сводка зависимостей

| Сосед | Для MVP ISMeta | Мок |
|---|---|---|
| Сервис распознавания | критически нужен | да, фиксированный JSON |
| API каталога/прайсов в ERP | критически нужен | да, фикстуры |
| LLM-шлюз с учётом токенов | нужен, упрощённо | да |
| Интеграции поставщиков | не нужен | — |
| Почтовый парсер | не нужен | — |
| RAG норм | не нужен | — |

## 7. Этапы и приоритеты разработки

> Подробно: [`specs/06-migration-plan.md`](./specs/06-migration-plan.md) (миграция кода), [`specs/07-mvp-acceptance.md`](./specs/07-mvp-acceptance.md) (критерии готовности), [`specs/08-stage1-dependencies.md`](./specs/08-stage1-dependencies.md) (граф зависимостей эпиков).

Оценок в человеко-днях нет — это следующий уровень планирования.

### Этап 0. Инфраструктура (реалистично 4–6 недель)

Пункты (после критики B4):
- создать `ismeta/backend/` и `ismeta/frontend/`;
- поднять отдельную PostgreSQL-БД для ISMeta;
- обновить `dev-local.sh` для двух стэков (ERP + ISMeta), общий postgres, общий redis;
- настроить CI с jobs для ismeta-backend и ismeta-frontend;
- определить версии Python/Django/Node.js (подтвердить текущие в проде);
- создать каркас трёх потоков документации `docs/ismeta/{dev,user,admin}/`;
- поднять в dev-окружении ДВА тестовых Workspace с разными UUID (early-test для multi-tenancy);
- создать базовый мок сервиса распознавания и мок ERP API;
- договориться о версиях LLM-провайдеров и провести первые cassette-записи.

### Этап 1. MVP «сметчик + автоматика + один LLM-инструмент»

Цель: конкретный сметчик (сами внутри команды играем роль) делает смету от начала до кнопки «Отдать в ERP» без обращения к прошлому интерфейсу (уточнение после критики B1: «сметчик» — это мы сами и внутренние QA на первом этапе, никакой реальной смёточной эксплуатации ERP до этого не было).

Работы:
- модели ISMeta (из списка в 3.1) + миграция логики работы matching, markup, excel_exporter из текущего `backend/estimates/`;
- Excel-цикл с row_id и hash;
- live-интеграция с API каталога/прайсов ERP;
- webhook-приёмник ISMeta + outbox-паттерн в ERP;
- фронтенд: основные экраны (список смет, редактор, диалоги наценок, импорт/экспорт);
- `@ismeta/widget` npm-пакет с `EstimateApiProvider`;
- ERP: endpoint «приёма snapshot сметы» (создаёт ContractEstimate), UI в ERP для принятия ДОП-решений;
- вынесение SpecificationParser в отдельное приложение `recognition/`;
- LLM-шлюз — упрощённый учёт токенов;
- **LLM-агент в MVP-составе: 2 инструмента, «Переподбор аналога по ТТХ»** (уточнение после критики B2 — этот один MVP-инструмент остаётся в MVP, мы договаривались);
- документация: dev + user + admin (стартовые версии);
- тестирование: cassette-tests для LLM, golden set из 10 смет, E2E через Playwright по основному сценарию, optimistic locking-тесты.

Не в MVP: остальные инструменты агента, публичный режим, RAG норм, почтовый парсер, защита «мозга», УКЭП, печатные формы, импорт из конкурентов, локализация кроме ru-RU.

### Этап 2. LLM-агент и публичный режим

- остальные инструменты агента (`get_estimate_summary`, `get_section`, `search_knowledge`, `get_supplier_prices`, позже `propose_change`/`apply_change`);
- ProductKnowledge batch-review workflow (сметчик раз в день подтверждает/отклоняет очередь новых знаний);
- публичный режим: URL, OTP, виджет ставки часа, JS-пересчёт, push заявки в ERP;
- дашборд метрик (базовый, без materialized views — после критики C4);
- расширение supplier integrations (≥ 2 интеграции);
- почтовый парсер MVP.

### Этап 3. Коробка и RAG

- упрощённый каталог внутри ISMeta для коробочных клиентов без ERP (после критики G2);
- адаптация дистрибутива: install.sh, мастер настройки, Admin Guide;
- первый коробочный пилот;
- RAG нормативных документов (pgvector, загрузка СНиП);
- Google Sheets как альтернатива Excel;
- long-term memory агента «Заметки сметчика».

### Этап 4+. Масштабирование

- SaaS multi-tenant с биллингом;
- параметризуемые печатные формы;
- УКЭП и электронная подпись;
- импорт из Гранд-Сметы/Турбосметчика;
- расширение на смежные отрасли;
- мобильный публичный просмотр.

### 7.5 Команда и профили (добавлено после критики F6)

## 8. Тестирование, метрики, документация

> Подробно: [`specs/11-metrics.md`](./specs/11-metrics.md) (метрики), [`specs/14-golden-set.md`](./specs/14-golden-set.md) (корпус эталонных смет), [`specs/13-release-process.md`](./specs/13-release-process.md) (релизы).

### 8.1 Тестирование

- **Unit** (pytest, vitest) — стандарт.
- **Cassette-tests** для LLM: записанные ответы, хранятся в репо, в CI воспроизводятся без реальных вызовов.
- **Golden set** 10 смет — эталоны для регрессии. Расширение до 30-50 — этап 2.
- **Определение «качество подбора» формулой** (после критики C1):
  - `matched` — строка, где сметчик принял авто-подбор без изменения;
  - `rejected` — строка, где сметчик заменил авто-подбор вручную;
  - `unmatched` — строка без предложения;
  - `Precision = matched / (matched + rejected)`;
  - `Recall = matched / (matched + rejected + unmatched)`;
  - Baseline фиксируется на первых 10 сметах golden set, без заранее объявленных цифр (после критики E6).
- **LLM-as-judge** — для чата агента (этап 2).
- **Реальные интеграционные прогоны** с настоящими LLM — на каждый PR, затрагивающий prompt/tool/pipeline подбора (а не раз в неделю, после критики E7). Для остальных PR — cassette.
- **E2E Playwright** по ключевым сценариям.
- **Optimistic locking** — отдельные тесты на конфликт.
- **Multi-tenancy isolation** — каждый тест прогоняется с двумя Workspace, утечка между ними = fail.

### 8.2 Метрики (дашборд в ISMeta, упрощённый)

| Метрика | Что даёт |
|---|---|
| Precision/Recall по тирам подбора | качество модели |
| Время от загрузки до готовой сметы | KPI продукта |
| Стоимость LLM-токенов на смету | экономика (см. 4.4) |
| Top-N не найденных позиций | что дополнять в каталоге |
| Top-N отклонённых подборов | что дообучать |
| Latency LLM-вызовов | health |
| Глубина очереди Celery | health |

Реализация — простые SQL-запросы и Recharts. Без partitioning, без TimescaleDB (после критики C4).

### 8.3 Документация

Три потока с этапа 1 (пользователь подтвердил «всю документацию сразу»):

- **Developer Docs** (`docs/ismeta/dev/`): архитектура, ER-диаграмма, OpenAPI-спецификации всех контрактов (3.5), как добавить LLM-провайдера, как добавить Tier подбора, как мигрировать схему.
- **User Guide** (`docs/ismeta/user/`): сценарии сметчика в markdown + базовый in-app onboarding (без видео в MVP, после критики C-замечаний по сложности — видео позже).
- **Admin Guide** (`docs/ismeta/admin/`): развёртывание, настройка LLM-провайдера, восстановление из бэкапа, замена API-ключей, IP allow-list для webhook'ов, rollback релизов.

## 9. Открытые риски и компромиссы

### 9.1 Subsection в MVP

В проде 0 Subsection. В MVP оставляем модель, скрываем в UI за флагом «расширенная иерархия». Если через год не пригодится — удаляем.

### 9.2 Точечные правки при новой версии

v_n+1 в ISMeta копируется с v_n без точечных ERP-правок. Переносит ли ERP-сметчик точечные правки в новый ContractEstimate — решает сам в ERP. Пользователь подтвердил (критика G1): «сметчик надёжный и опытный, жирного предупреждения достаточно». Принято.

### 9.3 Локальный LLM

Качество ожидается ниже, конкретных цифр не обещаем до бенчмарков. Clients pay this tradeoff consciously.

### 9.4 Экспорт «мозга» в коробку

Базовый снимок ProductKnowledge и `.md` уходит в коробку клиента. Возможно копирование клиентом в свою пользу. Юридическая и техническая защита — приложение Г (пользователь в критике F3 сказал «ничего страшного пока», фиксируем).

### 9.5 Webhook надёжность

Outbox + idempotent + polling (60 сек) = «at-least-once». Возможен короткий период расхождения. Для нашего домена терпимо. Если понадобятся строгие гарантии — Kafka.

### 9.6 Multi-tenancy «на бумаге»

Защита — early-test mode с двумя Workspace в dev-окружении с этапа 1 (после критики G4). Любая утечка — баг.

### 9.7 Распознавание — границы сервиса

Решено: отдельное Django-приложение внутри ERP. Если в будущем нужно выделить в микросервис — рефакторим.

### 9.8 Размер golden set

10 смет — это мало. План пополнения: каждую десятую реальную смету сметчика в эксплуатации добавляем в golden set. Расширение до 30-50 — этап 2.

### 9.9 Зависимость от ERP

В MVP ISMeta неотделима от ERP. Коробочная независимость — этап 3+ (после критики G2: это блокер для коробки, до него год и больше).

### 9.10 Автономность LLM-агента

В MVP — только предложение в чат, мутаций нет. В Этапе 2 — настройка «доверенных действий», но каждая мутация остаётся в AuditLog с пометкой «применено агентом», откат возможен.

### 9.11 152-ФЗ и данные в облачных LLM

Для российских корпоративных клиентов — необходимо на этапе коробки. В MVP для Августа — не критично (пользователь подтвердил). Отмечено в приложении Г как блокер коробочных продаж крупным компаниям.

### 9.12 Стоимость LLM в долгосроке

Раздел 4.4 фиксирует целевой ориентир 50 ₽ на смету. Превышение — повод оптимизации. Без учёта — риск неконтролируемых расходов.

## Приложение В. Список планов соседних сервисов

По договорённости (критика A1): для каждого соседа ниже нужно сделать отдельный детальный план. Этот список — чтобы не потерять.

### В.1 План сервиса распознавания (DocAI)

- **Цель:** PDF/Excel/изображение → структурированные items[].
- **Что должен содержать план:**
  - выделение `SpecificationParser` в Django-app `recognition/`;
  - OpenAPI для контракта `/recognition/parse/`;
  - схема результата JSON (items, document_meta);
  - кэш результатов по SHA256 файла (уже есть в `ParsedDocument`);
  - обработка ошибок (таймауты LLM, частичный парсинг, невалидный PDF);
  - интеграция с антивирусом;
  - базовый режим (PyMuPDF + один Vision LLM) и продвинутый (мульти-LLM, fine-tune) — разделены.
- **Приоритет:** критический для MVP ISMeta.

### В.2 План API каталога и прайсов в ERP

- **Цель:** публичный REST API к Product, WorkItem, PriceList, ProductPriceHistory, WorkerGrade.
- **Что должен содержать план:**
  - OpenAPI-спецификация всех эндпоинтов;
  - пагинация, фильтры, поиск;
  - outbox-таблица для webhook-событий;
  - фоновый воркер для доставки событий;
  - эндпоинт `/api/sync/events?since=...` для polling fallback;
  - rate limits для защиты от перегрузки.
- **Приоритет:** критический для MVP ISMeta.

### В.3 План LLM-шлюза (доработка `backend/llm_services/`)

- **Цель:** учёт токенов и стоимости, базовая квотация.
- **Что должен содержать план (MVP-часть):**
  - таблица `llm_usage` с полями tokens_in/out, cost, workspace, task;
  - обёртка вокруг каждого провайдера для автоматического учёта;
  - базовый дашборд «расходы по workspace».
- **Что вынесено в итерацию 2:**
  - квоты и лимиты на workspace;
  - алерты на превышение;
  - витрина по task_type и провайдерам;
  - автоматическое переключение между провайдерами при ошибках квоты.
- **Приоритет:** параллельный с MVP, минимальный вариант — к этапу 1.

### В.4 План RAG-базы нормативных документов

- **Цель:** индексация и поиск по СНиП/ГОСТ/ГЭСН/ТСН/МДС.
- **Что должен содержать план:**
  - выбор стека (pgvector в общем Postgres или Qdrant);
  - embedding-модель (многоязычная, учитывающая русский технический лексикон);
  - источники документов: открытые + платные (НОСТРОЙ), правовые аспекты;
  - чанкинг документов с учётом таблиц и формул;
  - API `/rag/search?query=...` для LLM-агента;
  - UI для админа по управлению корпусом.
- **Приоритет:** не для MVP ISMeta. Этап 2-3.

### В.5 План интеграций с поставщиками

- **Цель:** расширить `backend/supplier_integrations/` до 2+ интеграций.
- **Что должен содержать план:**
  - унифицированный адаптерный интерфейс;
  - 1-2 новые интеграции кроме Breez;
  - RFQ workflow end-to-end (запрос → email → ответ → привязка к смете);
  - веб-скрейпинг как fallback для поставщиков без API.
- **Приоритет:** после MVP.

### В.6 План почтового парсера

- **Цель:** приём и классификация входящих писем.
- **Что должен содержать план:**
  - IMAP-клиент + SMTP для отправки;
  - классификатор (LLM или правила): RFQ-ответ / общее / спам;
  - привязка писем к сущности RFQRequest;
  - UI для сметчика «входящие привязанные к моей смете».
- **Приоритет:** после MVP.

## Приложение В bis. Отложенные архитектурные решения (ревизия через 6 месяцев)

По итогам архитектурного ревью (см. [`docs/ARCHITECTURE-REVIEW.md`](./docs/ARCHITECTURE-REVIEW.md)) согласована стратегия «подморозить и вернуться». Эти пункты работают как есть для MVP, но помечены на пересмотр.

| Пункт | Решение сейчас | Триггер пересмотра |
|---|---|---|
| Redis HA (Sentinel/Cluster) | Один Redis, graceful degradation | Переход в production с > 10 сметчиков |
| EstimateSubsection в UI | Скрыт за feature flag | 6 месяцев эксплуатации — либо включаем, либо удаляем |
| Monolith Django vs микросервисы для recognition | Django-app в ERP | Нагрузка на recognition > 100 PDF/день |
| Next.js vs Vite | Next.js (как в ERP) | Оценка bundle size после этапа 1 |
| Event sourcing | CRUD + shadow domain_event таблица | После первого крупного клиента с compliance-требованиями |
| Multi-ERP на один Workspace | Не поддерживается (один Workspace = один ERP) | Первый холдинг-клиент |
| Separate service для LLM-шлюза | В ERP monolith | SaaS-запуск (этап 4) |
| Recognition: build vs buy | Build (своими силами) | См. [`docs/RECOGNITION-BUILD-VS-BUY.md`](./docs/RECOGNITION-BUILD-VS-BUY.md) после бенчмарка |
| CQRS для дашборда | Читаем из primary | Деградация p95 > 2s на /admin/metrics |
