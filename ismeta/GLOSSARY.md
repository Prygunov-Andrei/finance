# Глоссарий

Все термины, которые встречаются в документации ISMeta. Разделены на предметную область (смётное дело) и технические.

## Предметная область

### Смета
Документ, описывающий состав и стоимость работ и материалов для строительного/монтажного проекта. Обычно содержит иерархию: разделы → (подразделы) → строки-позиции.

Пример: «Смета на систему вентиляции корпуса А»: 5 разделов (воздуховоды, вентиляторы, кондиционеры, автоматика, пусконаладка), в каждом — десятки или сотни строк.

### Спецификация
Документ от проектировщика, описывающий состав оборудования и материалов для объекта, без цен. По сути — «что нужно купить и поставить». Приходит в ISMeta как PDF или Excel, ISMeta превращает её в смету, добавляя цены и работы.

### Сметчик
Сотрудник организации, готовящий сметы. В ISMeta он — основной пользователь. До автоматизации сметчик собирает смету вручную: пробивает каждую позицию спецификации по прайсам, ищет замены, корректирует цены. С ISMeta — получает автоматический подбор и подтверждает или корректирует.

### Позиция / строка сметы
Одна строчка в смете: наименование (что), модель (артикул), бренд, единица измерения (шт / м.п. / м²), количество, цена, сумма. В коде — модель `EstimateItem`.

### Раздел сметы
Верхний уровень иерархии. Обычно группирует работы по системам: «Вентиляция», «Кондиционирование», «Слаботочные системы».

### Подраздел сметы
Опциональный второй уровень. В реальных сметах сейчас не используется, оставлен для совместимости. В MVP UI скрыт.

### Работа (WorkItem, расценка)
Описание трудового процесса: что делает монтажник, сколько часов занимает, какая квалификация нужна. Пример: «Монтаж крышного вентилятора до 200 кг», артикул «1.42», 7.70 часов, грейд 4. Стоимость работы = `часы × коэффициент × ставка_грейда`.

### Товар (Product) / материал / оборудование
То, что закупается и устанавливается: вентиляторы, воздуховоды, кабель, кондиционеры. В коде — модель `Product`. Цены товаров живут в ERP (из счетов поставщиков или интеграций).

### Прайс-лист работ (PriceList)
Набор расценок на работы, актуальных на определённый период. Один активный прайс-лист за раз. Содержит WorkItem'ы с возможными override'ами часов и коэффициентов. Наш прайс на проде — 238 работ в одном активном прайсе.

### Грейд (Разряд монтажника)
Уровень квалификации рабочего, влияющий на ставку часа. У нас 5 грейдов: 500 / 650 / 800 / 950 / 1100 ₽/час. Работы в прайсе привязаны к конкретному грейду.

### Ставка грейда
Стоимость одного часа работы монтажника соответствующего разряда. Настраивается в ERP. При её изменении приходит webhook `worker_grade.rate_changed`.

### Прайс работ vs. каталог товаров
- **Прайс работ** — наш, редкий (238 позиций), меняется пакетом раз в год.
- **Каталог товаров** — внешние поставщики + наши счета, много (9498 позиций), меняется постоянно.

### Наценка
Разница между закупочной и продажной ценой. В ISMeta — трёхуровневый каскад: на смете в целом → на разделе (override) → на строке (override). Может быть процентом, фиксированной ценой, фиксированной суммой.

### Аналог
Товар, заменяющий отсутствующий или недоступный оригинал, с близкими техническими характеристиками. Пример: вместо вентилятора MOB2600/45-3a подставляем Systemair MOBS-063 с похожим расходом воздуха.

### ТТХ (технические характеристики)
Набор параметров оборудования: расход воздуха, мощность, напряжение, габариты, масса. Используются для поиска аналогов.

### Подбор работ (work matching)
Автоматический процесс, связывающий каждую строку сметы с подходящей расценкой из прайса работ. Реализован как 8-уровневый pipeline (Tier 0-7, см. [`specs/01-data-model.md §5`](./specs/01-data-model.md) и [`DOMAIN-GUIDE.md`](./DOMAIN-GUIDE.md)).

### Tier 0-7
Уровни алгоритма подбора работ в порядке убывания уверенности:
- **Tier 0 (Default)** — у товара уже установлена работа-по-умолчанию;
- **Tier 1 (History)** — история подборов (ProductWorkMapping);
- **Tier 2 (PriceList)** — fuzzy-поиск по работам текущего прайса;
- **Tier 3 (Knowledge)** — база знаний ProductKnowledge;
- **Tier 4 (Category)** — работы той же категории товара;
- **Tier 5 (Fuzzy)** — fuzzy-поиск по всему каталогу работ;
- **Tier 6 (LLM)** — LLM-вызов для сложных случаев;
- **Tier 7 (Web)** — LLM + web search для совсем нетипичных позиций.

### Confidence (уверенность)
Число от 0 до 1, показывающее степень уверенности в подборе. Каждый tier формирует свой confidence. В UI сметчик видит его как индикатор: >0.8 зелёный, 0.5-0.8 жёлтый, <0.5 красный.

### Версия сметы
Снимок сметы в определённый момент жизни. Изначально смета v1, после согласования с заказчиком — v2, v3, и т.д. У объекта за жизнь может быть 20-30 версий. В БД — `parent_version_id` ссылается на предыдущую, `version_number` инкрементируется.

### Передача сметы в ERP (transmission)
Процесс отправки готовой сметы в ERP для оформления договора. После передачи версия становится read-only. Webhook `contract.signed` от ERP возвращает id договора.

### ТКП (Технико-коммерческое предложение)
Документ, который мы отправляем заказчику — смета в красивой форме с нашим логотипом, описаниями, схемами. Генерируется в ERP из сметы, у каждого клиента свой шаблон. **Не в скоупе ISMeta.**

### Монтажная смета (MountingEstimate)
Смета, по которой будет работать бригада монтажников. Содержит только работы (без материалов), с нарезкой по бригадам или этапам. Генерируется в ERP из основной сметы. **Не в скоупе ISMeta.**

### Договор (Contract) и ContractEstimate
Юридический договор с заказчиком плюс снимок сметы на момент подписания. Хранятся в ERP. После подписания точечные правки делаются в ContractEstimate (расходные материалы, аналоги, допработы), ISMeta не видит этого.

### Допсоглашение (ДОП, ContractAmendment)
Дополнение к договору при существенных изменениях сметы. Создаёт новую версию ContractEstimate в ERP. В ISMeta — отдельная новая версия Estimate.

### Накопительная смета (AccumulativeEstimate)
Смета, отражающая фактически выполненные и оплаченные работы к определённой дате. Основа для актов КС-2. Живёт в ERP.

### Акт КС-2 / КС-3
Формы актов выполненных работ (стандарт РФ). Генерируются в ERP из накопительной сметы. **Не в скоупе ISMeta.**

### Поставщик
Организация, у которой мы закупаем оборудование. У каждого свой каталог с ценами. Интеграция с поставщиком — отдельный сервис в ERP (`supplier_integrations`).

### RFQ (Request for Quote)
Запрос коммерческого предложения у поставщика. Сметчик (или LLM-агент во 2-й итерации) формирует RFQ, отправляет поставщику, получает ответ с ценами, прикрепляет к смете.

### РРЦ (рекомендованная розничная цена)
Цена, по которой товар можно показывать публично (на сайте Августа), без раскрытия закупочной цены и маржи. Настраивается в ERP через `PublicPricingConfig`.

### База знаний (ProductKnowledge)
Коллекция правил «строка «вентилятор крышный X» → подходит работа Y». Накапливается по мере работы сметчика и LLM. Каждая запись имеет статус: pending (требует проверки), verified (подтверждена), rejected (отвергнута). Хранится в БД ISMeta + копируется в .md-файлы для ручной правки.

### Golden set
Корпус эталонных смет для регрессионного тестирования качества подбора. 10 смет в MVP, расширяется со временем. См. [`specs/14-golden-set.md`](./specs/14-golden-set.md).

## Технические термины

### Workspace
Изолированный «рабочий контекст» в ISMeta. В нашей инсталляции один Workspace = ERP Август. В коробочной инсталляции = один клиент. В SaaS (будущем) — один Workspace на организацию. Идентификатор — UUID.

### Multi-tenancy
Архитектурный паттерн, при котором одна инсталляция обслуживает несколько изолированных клиентов (tenants). У нас — через `workspace_id` на каждой таблице. В MVP пока один Workspace.

### Folder
Папка для группировки смет внутри Workspace. Может быть привязана к Object из ERP через `external_ref` (клиенты Августа) или быть свободной (коробочные клиенты без сущности «объект»).

### Object (объект строительства)
Сущность ERP: здание, сооружение, проект, к которому привязаны сметы. В ISMeta не хранится как отдельная таблица, только ссылка (`external_ref`) в Folder.

### Snapshot
Полный слепок сметы (с разделами, строками, ценами, подборами), отправляемый в ERP при кнопке «Отдать в ERP». См. [`specs/02-api-contracts.md §2`](./specs/02-api-contracts.md).

### Outbox pattern
Надёжная доставка событий: ERP пишет их в свою таблицу `outbox`, фоновый воркер пушит в ISMeta, ISMeta дедуплицирует по `event_id`. Плюс polling fallback на случай потери вебхуков.

### Webhook
HTTP POST от ERP к ISMeta с событием. Подписан HMAC, имеет уникальный `event_id` для идемпотентности.

### Idempotency
Свойство операции давать одинаковый результат при повторных вызовах. У нас реализовано через `Idempotency-Key` в API и `event_id` в webhooks.

### Optimistic locking
Защита от потери изменений при параллельном редактировании: каждая строка имеет поле `version`, клиент передаёт `If-Match: {version}`, сервер отклоняет (409) если версия не совпадает.

### Cassette (в тестах)
Записанный ответ LLM-провайдера, используемый для воспроизводимых тестов без реальных LLM-вызовов.

### Tool use (в LLM)
Паттерн, при котором LLM не отвечает напрямую, а «вызывает» описанный инструмент (функцию), получает результат и рассуждает дальше. Наш агент использует 2 инструмента в MVP.

### ReAct
Паттерн LLM-агента: Reason (подумал) → Act (вызвал tool) → Observe (получил результат) → повторил. Именно такой агент у нас.

### Golden set
Набор эталонных примеров для проверки качества. Не тренировочная выборка, а валидационная — мы не подгоняем под неё промпты.

### Precision / Recall
- **Precision** = `matched / (matched + rejected)` — «из предложенных подборов какая доля принята».
- **Recall** = `matched / (matched + rejected + unmatched)` — «из всех строк какая доля нашла принятый подбор».

### LLM-as-judge
Использование одного LLM для оценки качества ответа другого LLM. Применяется в регрессионном тестировании агента (этап 2).

### JWT (JSON Web Token)
Токен авторизации. У нас — подписанный ERP (RS256), принимаемый ISMeta. Короткий (15 минут), обновляется refresh-токеном.

### HMAC-подпись
Хэш-сумма тела запроса с общим секретом. Защищает webhook от подделки.

### pgvector
Расширение PostgreSQL для векторного поиска. Используется в RAG-базе норм (этап 3+).

### RAG (Retrieval-Augmented Generation)
LLM сначала ищет релевантные документы в базе, потом на их основе отвечает. Планируется для поиска по СНиП/ГОСТ в этапе 3+.

### CDC (Change Data Capture)
Механизм потоковой репликации изменений из БД. Резервный вариант для кеша Product, если webhook + TTL окажутся недостаточны.

### Semver (Semantic Versioning)
Схема версионирования MAJOR.MINOR.PATCH. Используем для API-контрактов (v1, v2) и самого приложения.

### pact-тесты
Contract-тесты между сервисами: consumer (ISMeta) описывает ожидания, provider (ERP) проверяется против них. Предотвращают поломку интеграции при изменении API.

### dogfood / dogfooding
Использование своего продукта командой разработчиков. В MVP — единственный сметчик = член нашей команды.
