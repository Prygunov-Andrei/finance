# 14. Golden set — корпус эталонных смет

**Версия:** 0.1. **Назначение:** как собираем, храним, пополняем и используем корпус эталонных смет для регрессии качества.

## 1. Зачем

- Регрессионный гард при изменении промптов, LLM-провайдеров, pipeline подбора.
- Baseline для метрик Precision/Recall (см. `07-mvp-acceptance.md §3.2`).
- Примеры для cassette-tests.
- Обучающая выборка при экспериментах с fine-tune.

## 2. Состав корпуса

### 2.1 MVP (этап 1): 10 смет

| # | Размер | Отрасль | Источник | Назначение |
|---|---|---|---|---|
| 1 | 20 строк | ОВиК — вентиляция | синтетическая | smoke-test, self-check |
| 2 | 50 строк | ОВиК — кондиционирование | синтетическая | базовый сценарий |
| 3 | 100 строк | ОВиК — комплекс | из 4 имеющихся в проде | первая реальная |
| 4 | 200 строк | ОВиК — сложный проект | из 4 имеющихся в проде | вторая реальная |
| 5 | 400 строк | СС — слаботочка | из 4 имеющихся в проде | разнообразие отрасли |
| 6 | 700 строк | ОВиК + СС | из 4 имеющихся в проде | четвёртая реальная |
| 7 | 1200 строк | ОВиК | синтетическая | нагрузочная |
| 8 | 2000 строк | ОВиК + СС | синтетическая от реальных | типовой размер |
| 9 | 4000 строк | ОВиК + СС | синтетическая от реальных | верхняя граница типового |
| 10 | 8000 строк | ОВиК + СС | синтетическая от реальных | стресс-тест |

Каждая смета в golden set описана:
- исходный файл (PDF или Excel);
- эталонная разметка items[] (то, что ожидаем от распознавания);
- эталонные подборы work_item_id по каждой строке (то, что сметчик бы согласился применить);
- эталонные подборы product_id с источниками (history / pricelist / manual);
- ожидаемые итоги (суммы по разделам).

### 2.2 Этап 2: расширение до 30 смет

- +10 смет разных размеров с фокусом на сложных строках (нестандартные ТТХ).
- +5 смет с «битым» распознаванием (специально плохой OCR) для проверки робастности.
- +5 смет с переподбором через LLM-агента (эталон сценариев чата).

### 2.3 Этап 3: расширение до 50-100 смет

- добавление смет из коробочных пилотов;
- разнообразие отраслей (если планируется расширение вне ОВиК/СС).

### 3.2 Метаданные (`metadata.yaml`)

```yaml

name: "ОВиК — малая вентиляция"

  precision: 0.85
  recall: 0.80
  pass1_duration_ms: 150
  llm_cost_rub: 2.50
```

### 3.3 Anonymisation

Реальные сметы из прода перед помещением в golden set:
- заменяется название проекта на обезличенное;
- удаляются реквизиты контрагентов (если были в содержимом);
- остаются технические поля — наименования оборудования, модели, количества.

Process: `scripts/anonymize.py` + ручной просмотр.

## 4. Процесс разметки

### 4.1 Кто размечает

- Первые 10 смет — команда ISMeta (роль dogfood-сметчика).
- Начиная с 30-й сметы — приглашённый эксперт-сметчик (из ERP Август), верификация.
- При коробочных клиентах — их сметчики размечают свои примеры (если готовы делиться).

### 4.2 Процедура разметки

1. Взять исходный файл (PDF или Excel).
2. Прогнать через сервис распознавания → `actual_recognition.json`.
3. Сравнить вручную с исходником; исправить ошибки → `expected_recognition.json`.
4. Создать смету в ISMeta из `expected_recognition.json`.
5. Запустить подбор работ (все tier'ы) и материалов.
6. Для каждой строки решить: ACCEPT / REJECT / CHOOSE_ALTERNATIVE.
7. Зафиксировать выбор в `expected_matches.json`.
8. Проверить итоги, зафиксировать в `expected_totals.json`.
9. Пройти PR review, добавить в репо.

### 4.3 Формат `expected_matches.json`

```json
{
  "estimate_id": "01-vent-20",
  "items": [
    {
      "row_id": "9b81...",
      "item_name": "Вентилятор крышный MOB2600/45-3a",
      "expected_work_item": {
        "work_item_id": "wi-503",
        "tier": "knowledge",
        "confidence_min": 0.85,
        "alternatives_acceptable": ["wi-504", "wi-505"],
        "rationale": "Монтаж крышного вентилятора до 200 кг подходит по мощности"
      },
      "expected_product": {
        "product_id": "prod-1234",
        "tier": "history",
        "price_source": "supplier_catalog",
        "alternatives_acceptable": ["prod-5678"]
      }
    }
  ]
}
```

## 5. Регрессионный прогон

### 5.1 Запуск

```bash
make ismeta-golden-run
```

Что делает:
1. Запускает ISMeta в test-режиме с cassette LLM.
2. Для каждой сметы в `golden/estimates/`:
   - создаёт Workspace, Folder, Estimate;
   - импортирует `expected_recognition.json` через mock-recognition;
   - запускает подбор работ + материалов;
   - собирает actual_matches;
   - сравнивает с expected_matches.
3. Формирует отчёт в `golden-report-YYYY-MM-DD.json`:
   - precision, recall, F1 по каждой смете и тиру;
   - latency Pass 1, Pass 2;
   - стоимость LLM (при реальном прогоне);
   - список deviations (где actual ≠ expected).

### 5.2 Критерии прохождения

- **Precision** per item не падает ниже зафиксированного baseline минус 5%.
- **Recall** не падает ниже baseline минус 5%.
- **Latency** не превышает baseline × 1.2.
- **Стоимость** (при реальном прогоне) не превышает baseline × 1.3.
- Новые deviations помечаются и либо становятся новым baseline (если легитимны), либо чинятся (регрессия).

### 5.3 Частота

| Прогон | Частота | LLM |
|---|---|---|
| Cassette | на каждый PR | не используется, только cassette |
| Реальный LLM | раз в неделю автоматически | используется |
| Реальный LLM, ручной | перед каждым релизом в production | используется |

## 6. Обновление baseline

### 6.1 Когда обновлять

- Улучшили промпт → precision вырос → новый baseline.
- Сменили LLM-провайдера → новый baseline (с комментарием в git).
- Обнаружили разметочную ошибку в expected → исправляем expected, baseline может не меняться.

### 6.2 Процедура

1. `make ismeta-golden-run` с реальным LLM.
2. Ревью результатов командой.
3. Если результаты лучше — `make ismeta-golden-update-baseline` (обновляет `baseline` в `metadata.yaml`).
4. Коммит + PR с пояснением «почему baseline изменился».

## 7. Cassette-tests на базе golden set

Cassette для каждой сметы живут в `ismeta/backend/tests/cassettes/golden/` (см. §3.1 и [`04-llm-agent.md §9.1`](./04-llm-agent.md)).

Обновление cassette — через `ISMETA_LLM_MODE=real make ismeta-golden-run REWRITE_CASSETTES=1`.

## 8. Anti-pattern: не переобучаться на golden set

- Golden set — **валидация**, не обучение. Промпты и pipeline не тюнятся «под него».
- Раз в квартал — сверяем golden set с реальными сметами, если синтетика стала «тренировочной» — обновляем.
- Изменения промпта после провала golden set — обосновываем, не подгоняем.

## 9. Владение и процессы

- **Владелец golden set**: LLM-инженер или QA-lead.
- **Ревьюер изменений**: любой сметчик из dogfood-группы.
- **Изменения** — через PR с обязательным ревью, не спонтанно.
- **Удаление сметы из корпуса** — только с обоснованием (например, стала нерелевантной из-за ухода с рынка).

## 10. Связь с коробкой

- Коробочному клиенту отдаётся снимок нашего golden set как обучающий пример.
- Клиент накапливает свой golden set по той же процедуре.
- Свой golden set клиента — его собственность, мы не копируем без разрешения.

## 11. Метрики корпуса

Следим за:
- размер корпуса (должен расти);
- распределение по размерам смет (не только маленькие или только большие);
- покрытие tier'ов (каждый tier срабатывает хотя бы на 2-3 сметах);
- время последнего прогона;
- время последней аннотации эталона.
