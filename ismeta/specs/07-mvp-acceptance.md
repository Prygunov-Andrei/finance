# 07. Критерии приёмки MVP и последующих этапов

**Версия:** 0.1. **Назначение:** измеримые критерии готовности, без которых нельзя объявить этап завершённым.

## 1. Общая рамка

Каждый этап CONCEPT.md (0, 1, 2, 3, 4) проходит формальную приёмку. Приёмка — это прохождение по списку критериев; если любой не пройден — этап не закрыт.

Критерии — трёх типов:
- **функциональные** (работает определённая фича);
- **качественные** (метрики на golden set, latency, стабильность);
- **процессные** (документация, тесты, релизная процедура).

## 3. Этап 1 — MVP

### 3.1 Функциональные

- [ ] Сметчик (член команды в роли dogfooder) создаёт смету из Excel-файла.
- [ ] Сметчик импортирует спецификацию через сервис распознавания (mock или реальный).
- [ ] Запускается подбор работ (все 8 tier'ов), Pass 1 (Tier 0-5) выполняется синхронно с прогрессом.
- [ ] Запускается подбор материалов.
- [ ] Сметчик редактирует смету: создаёт/изменяет/удаляет строки, разделы; меняет наценки на 3 уровнях.
- [ ] Сметчик экспортирует смету в Excel со скрытым `row_id` и hash; импортирует обратно с diff-предпросмотром.
- [ ] Сметчик экспортирует PDF.
- [ ] Сметчик создаёт новую версию сметы (copy + `source_item_id`).
- [ ] Сметчик открывает чат LLM-агента, использует `find-alternatives-by-specs` на конкретной строке.
- [ ] Сметчик передаёт смету в ERP (кнопка «Отдать в ERP»); ERP создаёт ContractEstimate.
- [ ] ERP подписывает договор; webhook `contract.signed` приходит в ISMeta; версия становится read-only.
- [ ] ERP передаёт события: `product.updated`, `worker_grade.rate_changed`, `pricelist.activated` — ISMeta обрабатывает корректно.
- [ ] Работает widget: встраивается в ERP-frontend через `@ismeta/widget`, авторизация через JWT.
- [ ] Работает аудит-лог: по любой сметной сущности видна история изменений.

### 3.2 Качественные

- [ ] **Golden set из 10 смет**: на всех 10 сметах подбор работ запускается до конца, хотя бы 50% строк получают match.
  **Процедура фиксации baseline'а:**
  1. Первый полный прогон golden set с реальным LLM — результат сохраняется как `baseline_v1`.
  2. Review командой: отметить сметы, где качество неприемлемо, исправить (либо разметку, либо pipeline), повторить прогон.
  3. Приемлемый результат — фиксируется в `ismeta/backend/tests/golden/baselines/v1.json`.
  4. С этой точки все PR сверяются с baseline'ом: регрессия = `precision` или `recall` падает >5%.
  5. На этом и последующих этапах baseline повышается только при обоснованном улучшении (документируется в PR).
- [ ] **Latency Pass 1** на смете 4000 строк — не более 90 секунд (target — 60).
- [ ] **Latency Pass 2** на смете 4000 строк — не более 15 минут.
- [ ] **Latency подсказки LLM-агента** (один вопрос, один tool-вызов) — P95 не более 20 секунд.
- [ ] **Стоимость LLM на смету** в среднем — не более 50 ₽.
- [ ] **Multi-tenancy isolation** — 0 утечек между двумя test-workspace'ами в E2E-прогонах.
- [ ] **Optimistic locking** — 0 потерянных изменений в тесте двух параллельных вкладок.
- [ ] **Uptime в staging за 2 недели перед cut-over** — не менее 99%.

### 3.3 Процессные

- [ ] Все OpenAPI-контракты сгенерированы из DRF и прошли валидацию.
- [ ] Pact-тесты между ISMeta и ERP проходят в CI.
- [ ] Cassette-тесты для LLM проходят в CI без реальных вызовов.
- [ ] E2E Playwright на основной сценарий проходит в CI.
- [ ] Покрытие unit-тестами backend ISMeta ≥ 70%.
- [ ] Покрытие unit-тестами frontend ISMeta ≥ 50%.
- [ ] Developer Docs: актуальны архитектура, ER-диаграмма, API-index.
- [ ] User Guide для сметчика: описаны все сценарии из 3.1.
- [ ] Admin Guide: описано развёртывание, бэкапы, восстановление, rollback релиза, замена API-ключей.
- [ ] Релизная процедура отработана минимум дважды на staging.
- [ ] Rollback-процедура отработана: откат релиза с обратной миграцией БД.

### 3.4 Что НЕ входит в MVP-acceptance

Чтобы не было двусмысленности:
- **EstimateSubsection в UI** — не обязательно, скрыто за feature flag.
- **Полный LLM-агент** — в MVP только два инструмента (`get_item`, `find_alternatives_by_specs`), чат работает только в режиме «подсказать аналог».
- **Публичный режим** — этап 2.
- **RAG норм** — этап 2/3.
- **Коробочная поставка** — этап 3.
- **Google Sheets sync** — backlog.
- **Three-way merge для Excel импорта** — simplified version в MVP, см. ADR-0016.

См. полный список упрощений в [`../docs/MVP-SIMPLIFICATIONS.md`](../docs/MVP-SIMPLIFICATIONS.md).

### 3.5 Business acceptance (кроме технических)

MVP считается готовым для внутренней эксплуатации, только когда:

- [ ] Dogfood-сметчик (член команды) подтверждает: «могу пользоваться этим вместо Excel».
- [ ] Цикл «спецификация → смета → передача в ERP» успешно прошёл 5 раз на реальных данных ERP Август.
- [ ] Время на смету 2000 строк в ISMeta измерено и сравнено с Excel-workflow.
- [ ] Собрана первая версия `docs/PILOT-RESULTS.md` с реальными цифрами.
- [ ] 3 pilot-клиента определены и проведены discovery calls (см. `../docs/GTM.md §3`).

### 3.6 Acceptance-критерий «MVP готов»

Объявляется закрытым, когда все чекбоксы 3.1–3.5 зачеркнуты **И** команда провела 2 недели эксплуатации в staging без критических инцидентов.

## 4. Этап 2 — LLM-агент и публичный режим

### 4.1 Функциональные

- [ ] LLM-агент имеет 7 инструментов: `get_estimate_summary`, `get_section`, `get_item`, `find_alternatives_by_specs`, `search_knowledge`, `get_supplier_prices` (+ зарезервированные propose_change/apply_change — но не в продакшене).
- [ ] Батч-ревью ProductKnowledge: сметчик видит очередь pending знаний, может bulk-verify/reject.
- [ ] Публичный режим: отдельный URL, OTP-регистрация посетителя, загрузка файла, статус, результат.
- [ ] Виджет публичного режима: ставка часа по грейдам, JS-пересчёт, РРЦ для материалов.
- [ ] Заявка посетителя пушится в ERP как событие + у ERP-сметчика появляется смета в списке.
- [ ] Дашборд метрик в админке (базовый): precision/recall по тирам, latency, LLM-стоимость, top-N ошибок.
- [ ] Вторая интеграция с поставщиками (помимо Breez).
- [ ] Почтовый парсер MVP: ответы поставщиков на RFQ привязываются к запросу.

### 4.2 Качественные

- [ ] **Golden set расширен до 30 смет**.
- [ ] **LLM-as-judge для агента** — оценка качества ≥ 4/5 на 80% сценариев.
- [ ] **Стоимость LLM на смету** с учётом всех инструментов агента — не более 80 ₽ (допуск роста из-за chat-интерфейса).
- [ ] Публичный режим: конверсия «загрузил → оставил заявку» — метрика фиксируется, baseline устанавливается.

### 4.3 Процессные

- [ ] User Guide расширен: работа с агентом, публичный режим.
- [ ] Admin Guide расширен: настройка публичного режима.
- [ ] Нагрузочные тесты на 10 параллельных чатов и 5 параллельных публичных сессий прошли.

## 5. Этап 3 — Коробка и RAG

### 5.1 Функциональные

- [ ] Упрощённый каталог внутри ISMeta (режим «без ERP»): загрузка CSV товаров, CSV прайса работ.
- [ ] Мастер установки: `install.sh` проводит через выбор LLM-провайдера, подключение каталога, генерацию ключей.
- [ ] RAG норм: загружен тестовый корпус (минимум 50 СНиП/ГОСТ), поиск работает через `/api/v1/rag/search`.
- [ ] LLM-агент использует RAG при релевантных вопросах («какая температура по СНиП для офисов?»).
- [ ] Пилот у партнёра: первая коробочная установка работает минимум 1 месяц.

### 5.2 Качественные

- [ ] Precision/recall на golden set коробочного клиента — не хуже чем на нашем.
- [ ] Uptime коробки у пилотного клиента — ≥ 99%.

### 5.3 Процессные

- [ ] Admin Guide полный, включая troubleshooting.
- [ ] Поддержка SLA согласована с пилотным клиентом.
- [ ] Документирована процедура обновления коробки.

## 6. Этап 4+ — SaaS и масштабирование

Acceptance-критерии формулируются по результатам этапа 3. Ориентировочно:
- Multi-tenant SaaS с биллингом работает с минимум 5 реальными клиентами.
- Аттестация 152-ФЗ (если требуется).
- Поддержка по SLA.

## 7. Регресс-тесты

Регрессионный набор растёт с каждым этапом. К моменту начала следующего этапа все регрессы предыдущего должны проходить в CI.

Состав:
- unit-тесты всех сервисов;
- integration-тесты всех API endpoints;
- E2E Playwright-сценарии;
- cassette-tests для всех LLM-флоу;
- pact-tests для всех межсервисных границ;
- golden-set прогон (раз в сутки в CI);
- isolation-tests для multi-tenant (каждый PR).

## 8. Нефункциональные требования к релизу

Любой релиз ISMeta проходит перед выкаткой в продакшен:
- Миграции БД — обратимы (есть `reverse_code` для всех RunPython).
- API-контракты — не содержат breaking changes (или major bump).
- Секреты — не в коммите.
- Лог-формат — совместим с агрегатором.
- Retention-политики данных — не нарушены (см. `12-security.md`).

## 9. Метрика «готов / не готов»

Скрипт `tools/acceptance-check.py` читает чекбоксы из этого документа и реальные состояния (тестовые прогоны, CI-статус, coverage-отчёт, статус Playwright-сценариев) и выдаёт зелёный/красный свет для релиза.
