# UX ревью ISMeta

**Дата:** 2026-04-15. **Роль ревьюера:** Senior Product Designer / UX Lead (синтетически). **Версия:** 0.1.

Ревью комплекса документов с позиции пользовательского опыта. Фокус — реальный workflow сметчика, cognitive load, usability heuristics, error prevention.

## 0. Методика

- Читаю CONCEPT, DOMAIN-GUIDE, specs/05 (Excel), specs/10 (public mode), specs/04 (LLM agent), EPICS.
- Применяю 10 эвристик Нильсена.
- Строю mental model сметчика. Ищу разрывы между документацией и реальным use case.
- Фокусируюсь на плотном UI редактора сметы — главной точке пользовательского взаимодействия.

## A. Что отсутствует как артефакты

Прежде чем критиковать решения — отмечу, что в комплексе **нет ни одного артефакта UX-дизайна**:

| Артефакт | Статус |
|---|---|
| Wireframes редактора сметы | ❌ |
| Wireframes публичного режима | ❌ |
| User flows (happy path и альтернативные) | ❌ |
| Information architecture | ❌ |
| Design system references | ❌ (shadcn/ui упомянут, но без гайдов) |
| Accessibility guidelines | ❌ |
| Empty states / error states каталог | ❌ |
| Keyboard shortcuts catalogue | ❌ |
| Loading states / skeleton states | ❌ |
| Onboarding in-app flow | ❌ |
| Responsive breakpoints | ❌ (desktop only) |
| Browser support matrix | ❌ |

**Это блокер для эпиков E9, E10, E11, E21.** Без макетов frontend-разработчик:
- либо додумает сам (получится фронт «как у всех»);
- либо прервётся и будет ждать дизайнера неделями.

**Что делать:**
- До старта E9 — провести design sprint: Figma-файл с core-экранами (список смет, редактор, диалоги подбора, чат агента, публичный режим).
- Design system — решить: делаем свой или используем shadcn/ui как есть.
- Минимум: lo-fi wireframes ключевых экранов.

## B. Mental model сметчика — несоответствие с архитектурой

### B1. Сметчик думает в терминах «смета, раздел, строка»

Мы добавили: Workspace, Folder, Version, Section, Subsection, Item, Characteristic.

**Cognitive load:** 7+ уровней абстракции. Сметчик будет теряться.

Что сметчик НЕ поймёт с первого раза:
- Разница между Estimate-as-entity и Estimate-version.
- Зачем Folder, если есть Object в ERP.
- Зачем Subsection (мы сами её скрыли — ура, но в UI она есть в settings).

**Что делать:**
- Скрыть Workspace в UI полностью (неявно, по JWT).
- Folder назвать «Объект» (если синхронизация с ERP) или «Папка» (без ERP).
- Версии сметы показывать **не как отдельные Estimate'ы**, а как временную шкалу одной сметы (Estimate-группы).

### B2. Версии — плоский список, а нужна шкала

В 30 версий сметы сметчик потеряется в плоском списке. Нужна:
- временная шкала (timeline) сверху редактора;
- метка статуса («отдана в ERP», «черновик», «текущая»);
- индикатор изменений относительно предыдущей (+N строк, изменений в M).

**Что делать:**
- Timeline-компонент в шапке редактора.
- Переход между версиями — один клик.
- Сравнение v_n vs v_n-1 — отдельная кнопка «diff» (см. раздел E).

### B3. Наценка трёхуровневым каскадом — непонятна

В DOMAIN-GUIDE объяснено, но в UI сметчик увидит:
- поле «наценка материала» на смете;
- поле «наценка материала» на разделе (которое переопределяет);
- поле «наценка материала» на строке (которое переопределяет оба).

Если он видит 35% — это глобальное, раздельное или строковое? Если глобальное 30%, а строка показывает 40% — это override или он настроил +10% дополнительно?

**Что делать:**
- Визуальная индикация источника наценки:
  - `30% (из сметы)` — серый текст.
  - `30% → 40% (override на строке)` — жёлтая подсветка.
- Hover с полной цепочкой: «глобально 30%, раздел 35%, строка 40% — применяется 40%».
- Кнопка «сбросить к родительской» на override'нутых полях.

## C. Главная боль: редактор сметы в 4000 строк

### C1. Плотность информации

Для средней сметы в 2-4 тыс строк, 20 столбцов — плотность ~80 000 ячеек на экране. Нильсен: «миниминизируй cognitive load».

Сейчас в специкации столбцов (`specs/05-excel-schema.md`):
- 20 столбцов.
- Из них 2 скрыты (row_id, row_hash) — хорошо.
- Остальные 18 — ВСЕ? На одном экране?

На экране 1920×1080 — помещается ~14 столбцов с приемлемой шириной. Остальные — либо мелкие, либо за скроллом.

**Что делать:**
- Column presets:
  - **Compact** (9 столбцов): №, Наименование, Ед, Кол-во, Закуп. мат, Закуп. работа, Прод. мат, Прод. работа, Итого.
  - **Extended** (15): + Модель, Бренд, Наценка мат %, Наценка работа %, Примечание, Поставщик.
  - **Full** (20): как сейчас, для power users.
- Сохранение preset'а per user (не только per estimate как в `ColumnConfigTemplate`).
- Quick switch через кнопку в toolbar.

### C2. Навигация по большой смете

Сметчик ищет «вентилятор» в смете из 4000 строк. Как?

Сейчас не описано: есть ли in-estimate search, jump to section, filter.

**Что делать:**
- `Cmd/Ctrl+F` — локальный поиск с highlighting (не браузерный).
- Sticky navigator — боковая панель со списком разделов, клик = скролл к нужному разделу.
- Breadcrumb навигация: `Смета №... > Раздел 3 > Строка 142`.
- Jump-to-row по `#ID`.

### C3. Редактирование inline — риски

4000 строк × 18 столбцов = 72 000 редактируемых ячеек. Один неверный клик = изменение. Сметчик это заметит когда?

Сейчас:
- Optimistic locking через `version` (есть).
- AuditLog (есть).
- Нет undo/redo на уровне UI.

**Что делать:**
- **Undo/redo stack** (last 50 operations) с `Cmd/Ctrl+Z`.
- **Autosave с indicator'ом** — «Сохранено 14:23» в углу.
- **Batch operations** (bulk) с preview: «вы выбрали 42 строки, применить наценку 25% ко всем?».
- **Toast notification** после каждого bulk-действия с кнопкой «Отменить» (30 секунд).

### C4. Virtualization — есть, но...

В `specs/09-dev-setup.md` указано TanStack Virtual. Хорошо.

Риски:
- **Tab navigation** между ячейками в virtualized table — сложно, не «из коробки».
- **Keyboard-driven** сметчик (привыкший к Excel) — шорткаты критичны.

**Что делать:**
- Keyboard shortcuts map (как в Excel): Tab/Shift+Tab для следующей/предыдущей ячейки, Enter для подтверждения, Escape для отмены, F2 для редактирования, Delete для очистки.
- Пользователю видима карточка шорткатов по `?`.

## D. LLM-чат — риски UX

### D1. Chat vs tools — непонимание режимов

В `specs/04-llm-agent.md` описан чат с tool use. Но в UI:
- Боковая панель с чатом постоянно? Или по кнопке?
- Кнопка «Найти аналоги» на строке — это вызов чата или прямой API?
- Если прямой API — почему используется тот же tool, что и агент?

**Что делать:**
- Два режима явно различаются:
  - **Direct action** (кнопка на строке): без чата, сразу показывает результат в модалке с top-5 аналогами.
  - **Chat mode** (боковая панель): свободный диалог, для нестандартных запросов.
- Направление пользователя к прямому action, чат — для продвинутых.

### D2. Latency LLM — «страшная пауза»

После вопроса сметчика агент думает 5-20 секунд. Пустое ожидание = плохой UX.

**Что делать:**
- SSE streaming (есть в spec). Показывать каждое слово по мере приёма.
- Индикация tool-calls: «🔍 Ищу аналоги в каталоге...» (видимо, что делает).
- Progressive disclosure: сначала краткий ответ, «Показать детали» — развёрнутый.

### D3. Hallucinations — нет defence UX

LLM может предложить несуществующий товар или неправильно подобрать работу. Сметчик может это пропустить.

**Что делать:**
- Каждое упоминание ID/artиcул'а в ответе агента — **кликабельное**, переход к сущности для проверки.
- «Источник» в tooltip'е: «найдено в каталоге ERP, обновлено 5 мин назад».
- **Disclaimer** на первом открытии чата: «AI-ассистент может ошибаться, проверяйте важные решения».

### D4. Cost visibility

Сметчик не знает, сколько стоит его запрос. Бюджет LLM есть (в админке), но сметчик не видит.

**Что делать:**
- Мини-индикатор «использовано 30% месячного лимита» в углу чата — не вторгается, но даёт awareness.
- При приближении к лимиту — warning: «осталось 5 запросов на этом тарифе».

## E. Версии и diff — слабое место

### E1. Как сравнивать v_n vs v_n-1?

Сценарий: заказчик попросил перерасчёт, сметчик сделал v2, теперь хочет показать заказчику «что изменилось».

Сейчас: ничего в UI не описано.

**Что делать:**
- Side-by-side diff view: две колонки, старая и новая версия, изменения подсвечены.
- Три типа diff'а:
  - **Structural**: добавились/удалились разделы, строки.
  - **Quantitative**: изменилось количество, цена, наценка.
  - **Qualitative**: изменилось имя, модель.
- Export diff в PDF с tracked changes для отправки заказчику.

### E2. Ветвление не поддерживается — проблема для multi-client

Сметчик готовит три варианта (дорогой / средний / дешёвый) для одного заказчика. Сейчас — линейное версионирование.

**Что делать:**
- **Short-term:** копирование сметы как базы новой — есть (`create_from_estimate` в ERP). Но UX не описан.
- **Long-term:** branching (как в git), merge — backlog.

### E3. Сохранение промежуточных версий — manual или auto?

Сметчик правит смету 3 часа. Создаётся ли auto-snapshot?

Сейчас не описано.

**Что делать:**
- Auto-draft раз в 5 минут (в отдельном пространстве, не плодим версии).
- Manual «save as version» — явная кнопка с именованием.

## F. Excel-цикл — hidden complexity

### F1. Путаница «что в Excel, что в UI»

Сметчик экспортировал Excel, поправил. Вернулся в UI. За это время через него же прошли обновления (webhook `product.updated` меняет кеш).

Что он увидит при импорте?

Сейчас описано через three-way merge (data-model.md, excel-schema.md). Но в UI:
- Как это объяснить сметчику, чтобы он не запутался?
- «Столбец имя в Excel: «Вентилятор XX», в БД: «Вентилятор XX v2» — выбери». Повторить 200 раз.

**Что делать:**
- Bulk-решение: «применить все изменения из Excel, отметить конфликты для ручного решения».
- Summary-карточка: «+15 строк, -3, изменено 42, конфликтов 5» — сметчик видит objem.

### F2. Feedback при экспорте

Сметчик нажал «Экспорт в Excel». Файл 4000 строк генерируется 10 секунд. Что он видит?

**Что делать:**
- Прогрессбар.
- Notification: «Файл сгенерирован, открыть?».
- Автоматическое открытие через system default (если пользователь включил).

## G. Публичный режим — UX специфика

### G1. Посетитель — не сметчик, cognitive load иной

Специалист OK с 18 столбцами. Посетитель — нет.

Спецификация `specs/10-public-mode.md` описывает функционал, но не упрощение интерфейса.

**Что делать:**
- Режим для публичного — радикально упрощённый:
  - вся смета — одна вертикальная таблица с 5 столбцами;
  - вместо материала/работы split — одна сумма;
  - виджет настройки ставки часа внизу.

### G2. Виджет ставки часа — риск frustration

Виджет пересчитывает мгновенно. Но:
- Посетитель не знает, что такое «грейд 3». Надо объяснить.
- Изменил ставку, увидел +500 000 ₽ — что делать?

**Что делать:**
- Tooltip на грейде: «Монтажник 3 разряда — опытный рабочий».
- Default-значения с объяснением: «мы предлагаем ставку 960 ₽/ч, это рекомендуемая розничная; вы можете указать свою».
- Сравнительная метрика: «изменение ставки на 100 ₽ увеличивает смету на 4%».

### G3. Post-OTP flow — drop-off риск

Шаги: email → OTP → загрузка → ожидание (1-3 минуты) → результат.

На каждом шаге — риск, что посетитель закроет вкладку.

**Что делать:**
- Email при готовности: «Ваш расчёт готов, откройте по ссылке».
- Save progress на каждом шаге — вернуться и продолжить.

### G4. Callback form — избыточные поля?

Форма заказа звонка: email (есть), имя, телефон, время, комментарий.

Для одного звонка — 5 полей = низкая конверсия.

**Что делать:**
- Минимум: телефон + кнопка «перезвонить». Всё остальное — опционально.

## H. Accessibility

Нигде в документации не упомянут. Предполагаем: dekstop-only (да), но люди с disabilities?

**Что критично:**
- **Keyboard navigation** — есть план, но не описан (см. раздел C4).
- **Screen readers** — DRF+Next.js с shadcn/ui из коробки не accessible. Нужна explicit работа: ARIA labels, focus management.
- **Color contrast** — WCAG AA minimum (4.5:1 для text).
- **Font size** — минимум 14px.
- **Dark mode** — опция (правильно, уже решили).

**Что делать:**
- `docs/A11Y.md` с уровнем соответствия (WCAG 2.1 AA).
- CI-job с axe-core или Lighthouse для проверки.
- Пункт в PR-checklist: «accessibility не ухудшилась».

## I. Error handling — UX stories

### I1. Ошибки сейчас «технические»

RFC 7807 (Problem Details) — хорошо для машин, плохо для людей.

`{"type":"...","title":"Validation failed","detail":"quantity must be positive"}` → пользователь увидит «Validation failed»?

**Что делать:**
- Error message layer на фронте: мэппинг `type` URI → человекочитаемое сообщение.
- Контекст ошибки: «не удалось сохранить количество в строке 142: должно быть положительным числом».
- Action button: «Вернуться к строке 142».

### I2. Network errors — забытый сценарий

Сметчик правит в редакторе. Интернет пропал на 30 секунд. Что?

**Что делать:**
- Detection через `navigator.onLine` + ping.
- Toast: «Соединение потеряно, изменения сохранятся локально».
- Auto-retry при восстановлении.
- При длительном offline — явный «offline mode» с ограничениями.

### I3. Concurrent editing — User knows what?

`409 Conflict` из optimistic locking — сметчику надо что-то показать.

**Что делать:**
- Модалка: «Эта строка была изменена в другой вкладке. Показать изменения / Перезаписать моими / Отменить правку».
- Diff визуализация: «ваши изменения vs свежая версия».

## J. Onboarding пользователя

### J1. «Пустое состояние» нового сметчика

Новый сметчик открывает ISMeta. Что видит?
- Список смет пустой.
- Что делать дальше?

**Что делать:**
- Empty state с CTA: «Создать первую смету» / «Импортировать из Excel» / «Загрузить PDF спецификации».
- Short video / interactive tour (15-30 сек) с тремя шагами.

### J2. Прогрессивное раскрытие фич

У ISMeta много features. Показывать сразу все — overwhelming.

**Что делать:**
- Feature tips: при первом попадании в экран — tooltip «Это X, здесь вы можете Y».
- Дополнительные возможности по мере необходимости («Вы создали 10 смет, попробуйте создать Folder для группировки»).

### J3. Документация в контексте

Пункты User Guide в `docs/ismeta/user/` — это отдельная страница. Сметчик в момент работы на неё не зайдёт.

**Что делать:**
- Context-sensitive help: "?" рядом с каждым диалогом, открывает snippet из User Guide.
- Searchable command palette (Cmd+K): «как сделать X» → подсказка + прямое действие.

## K. Локализация и интернационализация

Концепция: только русский язык в MVP. OK.

Но:
- Даты: формат DD.MM.YYYY vs MM/DD/YYYY vs ISO.
- Числа: разделитель тысяч (пробел в РФ, запятая в US, точка в DE).
- Валюта: ₽ перед или после числа.
- Pluralization русского: «1 позиция», «2 позиции», «5 позиций».

**Что делать:**
- next-intl заложен. Зафиксировать dictionary как source of truth.
- Все строки — через `t('key')`, никаких inline-текстов.
- CI-чек: нет hardcoded Russian/English.

## L. Mobile — сознательно не делаем, но...

Сметчик работает за desktop. Подтверждено.

Но:
- **Посетитель публичного режима** — часто с мобильного (маркетинг-канал).
- **Клиент, открывающий PDF-смету** — с мобильного.

Публичный режим на мобильном — must.

**Что делать:**
- Публичный режим: responsive до 320px.
- Редактор: gracefully degrade. На мобильном — read-only view с notice «для редактирования откройте на компьютере».

## M. Micro-interactions и polish

Детали, которые делают продукт «не корявым»:

- **Inline валидация** ввода — не только на отправке формы.
- **Debounced autosave** с visual feedback.
- **Skeleton loading** вместо пустых экранов.
- **Smooth transitions** между состояниями.
- **Hover tooltips** с useful info (не просто повтор текста).
- **Focus states** видимы (не только для keyboard users).
- **Success/error toasts** с правильной продолжительностью (success 3s, error дольше).
- **Undo via toast**: сразу после action — «Удалено 5 строк. Отменить» (30 секунд).

Ничего из этого не в спецификациях.

**Что делать:**
- Design system doc с паттернами этих micro-interactions.
- Component library (shadcn/ui) — проверить, что всё соответствует.

## P. Приоритетный UX-roadmap

### P1. Блокирующее E9 (редактор)

1. **Design sprint:** Figma с core-экранами (список смет, редактор с плотным гридом, сайдбар навигатор, toolbar).
2. **Column presets:** compact/extended/full.
