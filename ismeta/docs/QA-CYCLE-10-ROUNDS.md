### #45 🟡 UX — длинное наименование перетягивает ширину столбца

**Цитата PO:** «Если длинное наименование (и я его еще удлинняю), то столбец становится все шире, а остальные сужаются. Выглядит ужасно. В какой то момент должен произойти перенос текста, или скрытие части.»

**Симптом (скрин):** строка 1 «П1/В 1-Приточно - вытяжная установка L= 755/655м 3 / ч Pc=3 0 0 Па LM DUCT Q 40-20 комплектно со см. узлом, пластинчатым рекуператором и стандартным комплектом автоматики» — 180+ символов в одной строке, таблица растягивается далеко за viewport, численные столбцы сжаты.

**Корневая причина:** в `items-table.tsx` колонка «Наименование» не имеет `max-width` / `white-space: normal`. `EditableCell` рендерит одной строкой. TanStack Table auto-size по content.

**Решения (от простого к продвинутому):**
1. **Hotfix** (15 мин): CSS `max-width: 500-600px` + `white-space: normal` + `word-break: break-word` для колонки name. Текст переносится, столбец не растягивается. Остальные получают sensible ширины.
2. **Resize by drag** (~0.5 дня): TanStack Table supports `columnResizing`. Добавить resize-handle между заголовками, state в компоненте.
3. **Persist resize** (~0.5 дня): localStorage по `estimate_id` или `user_id`. Либо backend-поле user_preferences.column_widths.

**Вопросы PO (из сообщения):**
- «Насколько сложно регулировать ширину перетягиванием?» → **средне**, 0.5 дня Фёди (TanStack Table native support).
- «Фиксировать между сессиями?» → **просто**, localStorage 1-2 часа; либо backend user_preferences — 0.5 дня (migration).

**Severity:** 🟡 minor UX (не блокер, функционально работает). Но при 150+ позициях мешает читать.

**Root cause:** **R28** (новое) — колонка name без max-width и переноса.

**Решение PO:** вариант C — перенос + resize + persist.

**Статус:** ✅ закрыт. UI-08 замержен в main @ `f2b01cc` (2026-04-23).

### #50 🔴 R29 (новое) — разделы из Recognition не создаются в смете

**Цитата PO:** «Распознано — 141 (8 разделов) — так было написано в диалоге распознания. Однако в смете — нет ни одного раздела!»

**Симптом:** recognition вернул `sections[]` в ответе (UI-диалог их показал), а при создании сметы все позиции легли в один раздел по умолчанию.

**Гипотеза:** баг интеграции frontend → backend. Либо mapping `item.section_title → estimate_sections` не работает, либо frontend игнорирует sections[] из parse-ответа.

**Приоритет:** 🔴 блокер — ручное разбивание на разделы обессмысливает half of UI-09.

**Диагностика (2026-04-23):** recognition вернул 8 section_name'ов в items → ismeta-backend создал 8 `estimate_section` в БД (проверено: curl `/sections/` возвращает все 8) → **frontend mutation.onSuccess после PDF-upload не инвалидирует `["estimate-sections", estimateId]`**. react-query держит старый кэш (пустой, от первого открытия страницы) → UI рисует 0 разделов пока пользователь не F5.

**Fix:** добавить `qc.invalidateQueries({ queryKey: ["estimate-sections", estimateId] })` в onSuccess обоих импортов (PDF и Excel — симметрично).

**Файлы:** `frontend/components/estimate/pdf-import-dialog.tsx:105`, `import-dialog.tsx:83`.

**Статус:** ✅ закрыт. Коммит — см. git log (2026-04-23).

### #51 🔴 R30 (новое) — попарные строки многострочных имён: 4 разных поведения

**Цитата PO:** «В Спецификации было: Строка 1 "Дефлектор Цаги" + Строка 2 "на узле прохода". Распозналось: Строка 1 "Дефлектор Цаги" ВЕРНО, Строка 2 "Дефлектор Цаги на узле прохода УП1" — НЕ верно... Ключ к решению как будто лежит в плоскости понимания того, что у реальных товаров всегда есть количество. А у остатков строки — количества никогда нет.»

**Четыре разных поведения на одной PDF:**
- **Строки 1-2, 3-4** — склеивание с захватом имени соседа (Цаги «прилепился» к строке «на узле прохода УП1»).
- **Строки 58-77** — аналогичное склеивание для пар «Клапан…» + «с решеткой».
- **Строки 78-85** — вторая строка остаётся отдельной как «с решеткой» (не идеально, но проще править).
- **Строки 86-105** — вторая строка **полностью отброшена** (потеря продолжения имени «круглый морозостойкий»).

**Ключ PO:** «у настоящего товара всегда есть qty, у продолжения имени — никогда».

**Root cause:** текущий парсер определяет «это отдельная позиция» по наличию в строке номера позиции / названия, но не проверяет quantity как критерий. Промпт LLM тоже не использует этот signal жёстко.

**Решение (план):** правило пост-обработки + усиление промпта — если у распознанной строки `qty=0 AND unit=""` AND название начинается со строчной буквы ИЛИ с союза/предлога («с», «на», «круглый», «под») — это **продолжение** предыдущей строки, merge в `name`. **Не создавать** item.

### #52 🔴 R31 (новое) — потеря позиций в хвостах страниц (криминал)

**Цитаты PO:**
> «После строки 28 на странице 2 Спецификации было 4 позиции Воздуховод… В Смете все 4 позиции просто потеряны! И Пункт 29 — это уже с третьей страницы. Как обезопаситься от этого?»
> «…первые 5 позиций этих воздуховодов система просто пропустила (снова эти пропущенные позиции последние на странице!)»
> «Две предпоследние позиции сметы Противопожарная изоляция и Огнезащитная клеящая смесь — полностью утеряны.»

**Подсчёт потерь:** ~13 позиций (4 + 5 + 2 + 2). Совпадает с дельтой 153 → 141 минус 1 ложный item (#53).

**Root cause (гипотезы):**
- bbox-extractor обрезает низ страницы (зона штампа ЕСКД) — последние rows теряются.
- LLM в пределах одного call per page обрабатывает rows но дропает хвост из-за token-budget или формата output.
- sticky parent имя «перетекает» через page-boundary и затирает позиции.

**Идея PO (safety-net):** «отправлять в LLM страницу PDF с вопросом — сколько позиций на странице (подсказка: у настоящей позиции всегда есть количество)? и сравнивать с распознанным. Пропуск позиций — криминал!»

**План:** (а) диагностика на spec-ov2 — какие страницы теряют позиции и почему (rotation? footer?); (б) добавить LLM-count per page в отдельный lightweight call + cross-check + retry через multimodal; (в) вернуть из парсера `pages_summary: [{page, expected_count, parsed_count}]` для UI.

### #53 🟠 R32 (новое) — подразделы Спецификации распознаются как товары

**Цитата PO:** «Строка 106 — система создала нам товар "Фасонные изделия к вентиляторам ПДВ", и поставила произвольное количество — 1 шт. Но это просто подраздел в Спецификации (нет количества в Спецификации!).»

**Root cause:** промпт/парсер берёт строку без qty и принудительно ставит `qty=1`. Нет правила «no qty + no unit + не содержит технического артикула = подраздел/заголовок».

**Связь:** та же логика что и R30 (`qty=0 AND unit=""` → не item).

### #54 🟠 R33 (новое) — примечания «+10%» потеряны

**Цитата PO:** «В строках 5-12, 29-32 в Спецификации было примечание "+10%", в Смете — нет примечаний.»

**Root cause:** последняя колонка «Примечание» не всегда попадает в bbox extraction ИЛИ LLM-промпт её не указывает как обязательное поле OR mapping на `tech_specs.comments` рвётся.

**План:** проверить что bbox-extractor читает всю ширину страницы включая правый столбец; усилить промпт с явным указанием обрабатывать колонку «Примечание» → `comments`.

### #55 🔴 R34 (новое) — sticky-name «Решётка» поглощает чужие позиции (строки 124-131, 136-140)

**Цитаты PO:**
> «Строка 123 "ПН2-4,5-Решетка" — Все верно, но ниже, строки 124-131 система почему-то решила, что это тоже Решетки, в то время как строка с решеткой была только одна, а далее уже пошли Воздуховоды.»
> «Строка 136-140 — система решила почему-то что это снова Решетка… хотя это опять были Воздуховоды.»

**Root cause:** sticky parent name (правило что имя с предыдущей строки переносится вниз если текущая строка — variant/модификация) работает слишком агрессивно. Для одиночной «Решётки» не должно быть sticky-наследования — нужен счётчик применений sticky или проверка схожести follow-up строк.

### #56 🟡 R35 (новое) — склейка слов без пробела: «Защитныйкозырек»

**Цитата PO:** «Строка 111-113 — В наименовании "ПД1,2,3-Защитныйкозырек с сеткой" слова "Защитныйкозырек" — потерян пробел — а это два разных слова.»

**Root cause:** kerning-обработка (R24) склеивает токены через `gap < font_size * 0.3`. Для некоторых PDF-фонтов порог слишком агрессивный.

**План:** в DEV-BACKLOG — добавить post-process через словарь разделения общих русских слов (низкий приоритет, ручная правка быстрая).

### #57 🟡 R36 (новое) — непоследовательный разделитель pos/name: пробел vs дефис

**Цитата PO:** «Я заметил, что в некоторых случаях система сцепляет текст из первого столбца Спецификации (например "ВД1") с наименованием через пробел, а в некоторых случаях через "-".»

**Root cause:** prompt/normalizer не нормализует разделитель. В PDF сам разделитель может отличаться (пробел/дефис/тире).

**План:** в DEV-BACKLOG — единое правило в промпте «используй дефис без пробелов между pos и именем» ИЛИ «пробел» (PO выберет).

### Ключевой fix: `_merge_series_parent_into_children`

Новая функция в `recognition/app/services/pdf_text.py` (после `_merge_continuation_rows`). Триггер STRICT:

1. Parent row: name непустой, qty/pos/unit/mfr/brand пусты, НЕ section heading (model может быть)
2. 0+ continuation rows: name непустой, ВСЁ остальное (включая model) пусто
3. ≥2 suffix-children подряд: `cells.qty непустой` И `cells.name matches ^(n=\d+сек|Ду\d+|Ø\d+...)$`

Действие: для каждого child `cells.name = parent_full + " " + child.name` (с dash-rule word-break). Если child без model — наследовать parent.model. Parent+continuation rows удаляются (LLM их всё равно бы не эмитировала без qty).
