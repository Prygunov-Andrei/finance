### #1 🟠 major — PDF import: непрозрачная крутилка, пользователь не видит что происходит

> «Можем ли мы как-то дополнительно визуализировать работу по распознаванию? Основываясь на каких-то реальных данных? Сейчас у нас крутится кружочек и идут секунды, минуты, но что происходит — мы реально не понимаем — это бесит Пользователя.»

**Суть:** при загрузке PDF стадия `uploading` в `pdf-import-dialog.tsx` (line 27 `Stage = "choose" | "uploading" | "result"`) показывает простую крутилку в течение 15-60 секунд без каких-либо реальных индикаторов прогресса. Пользователь не понимает: обрабатывается ли его PDF, на каком этапе, сколько ещё ждать.

**Что реально делает Recognition под капотом:**
1. Render PDF → PNG (PyMuPDF) — ~0.2 сек на страницу
2. Classify LLM Vision — ~2.5 сек на страницу
3. Extract items LLM Vision — ~2.5 сек на страницу (если specification)
4. Deduplicate + save — ~0.5 сек

Итого: `1.5 + pages × 5` сек. На 4 страницах ~22 сек, на 30 страницах ~150 сек.

**Решение (два варианта в backlog):**

**Вариант A — Honest estimate + fake stages (quick-win, ~4 часа):**
- Новый endpoint `POST /v1/probe` в Recognition: принимает PDF, PyMuPDF считает страницы, возвращает `{pages_total}` за ~0.5 сек.
- ISMeta frontend: перед upload делает probe → показывает `«4 страницы, ~22 секунды»` + честный progress bar по времени + rotating hints («Рендеринг страниц», «LLM анализирует страницу 2 из 4», «Извлекаем позиции», «Дедупликация»).
- По окончании реальные цифры из ответа: `{pages_processed}/{pages_total}, items={N}, errors={M}`.
- Исполнители: IS-Петя (probe endpoint) + IS-Федя (UI).

**Вариант B — SSE streaming (1-2 дня, полноценный real-time):**
- Recognition emit событий `page.classified` / `page.extracted` по ходу.
- ISMeta backend держит SSE соединение к клиенту.
- Frontend рисует живой прогресс: «Страница 2 из 4 → 7 позиций найдено».
- Эпик: `E-PDF-STREAM`. Делать когда станет критично (длинные PDF 30+ страниц) или на polish перед prod релизом.

**Приоритет:** major UX, но не блокер — полный флоу проходит. **Делаем вариант A** (после demo и фикса более критичных багов — если найдутся). Вариант B — в backlog.

**Файлы:**
- `recognition/app/api/parse.py` (добавить `/v1/probe`)
- `recognition/app/services/pdf_render.py` (использовать `fitz.open` + `len(doc)`)

### #3 🔴 blocker архитектуры — Recognition игнорирует text layer PDF

> (Анализ golden PDF от Андрея: `ismeta/tests/fixtures/golden/spec-ov2-152items.pdf`, 9 страниц A3 landscape, 152 позиции заявлено.)

**Открытие:** PyMuPDF `get_text()` на всех 9 страницах возвращает **реальный текст** (462-1972 символа на странице), структурированный как таблица спецификации:
```
Дефлектор Цаги
ф355-оц-...
250х100
м.п.
6
300х150
м.п.
115
```

Это означает: PDF создан в CAD/Visio/AutoCAD (не скан), **текст вытаскивается напрямую без LLM** — мгновенно, бесплатно, с ~99% точностью.

**Почему текущий Recognition не использует это:**
- `SpecParser.parse` в `recognition/app/services/spec_parser.py` безусловно делает `render_page_to_b64()` → `POST OpenAI Vision`.
- Никакой проверки «есть ли text layer?» — скипаем prerequisite, идём в дорогой Vision.

**Последствия:**
- **Стоимость:** ~$0.01-0.05 per 10-страничная спецификация (OpenAI call'ы).
- **Скорость:** 88 сек на 19 страниц (5 сек/страница × 2 LLM call'а).
- **Качество:** 6/152 ≈ **4% recall** в прошлом прогоне (на другом PDF из 19 страниц — возможно с некоторыми сканами, но на этом 9-страничном text-layer PDF ожидаем тот же провал т.к. пайплайн одинаковый).

**Правильное решение — hybrid парсер:**

```python
# recognition/app/services/spec_parser.py
async def _process_page(self, doc, page_num):
    page = doc[page_num]
    text = page.get_text()              # шаг 0: проверить text layer
    if len(text.strip()) >= 100:         # есть text layer
        items = self._parse_text_layer(page)  # новый метод через get_text("dict") → bbox → rows
        # LLM только для нормализации имён (опционально, batch)
    else:
        # fallback: текущий путь LLM Vision для сканов
        page_b64 = await run_in_threadpool(render_page_to_b64, doc, page_num)
        ...
```

**Реализация text layer parsing:**
- `page.get_text("blocks")` возвращает список `(x0, y0, x1, y1, text, block_no, block_type)` — уже bbox для строк.
- Группировать по `y0` (строка таблицы), сортировать по `x0` (колонки).
- Определить колонки: обычно это `наименование / марка / ед.изм. / кол-во / примечание`.
- Отфильтровать заголовки/нижние колонтитулы (по y-координате или keywords «Спецификация», «лист», «Изм.»).

**Для golden set уже понятны разделы** (9 страниц: общеобменная жил, общеобменная МОП, воздуховоды+изоляция, противодымная вентиляторы, противодымная подпор, клапаны типовые, клапаны OKL, фасонные к ПДВ, воздуховоды по сечениям) — отличная основа для acceptance-теста.

**Плюсы hybrid-подхода:**
- 90% реальных спецификаций (созданных в CAD) → мгновенный парсинг, 99% точность, нулевая стоимость LLM.
- 10% сканов → текущий LLM Vision путь.
- LLM Vision становится **fallback + validator**, а не primary.
- Снимает проблемы #1, #2 (c), #2 (a), #2 (b) одним ударом — они все про Vision, которого не будет на большинстве PDF.

**Что делать:**
1. IS-Петя — новый эпик `E15.03 Hybrid text-layer + Vision parser`.
2. Фаза 1: добавить `_parse_text_layer()` в SpecParser, логика выбора пути (heuristic: `len(text) >= threshold`).
3. Фаза 2: тесты на golden set `spec-ov2-152items.pdf` с метрикой precision/recall (цель: ≥85% recall, ≥95% precision).
4. Фаза 3: оптимизация структурированного парсинга (группировка колонок, определение section_name).
5. В проме Vision временно поправить (#2): gpt-4o + max_tokens=16000 + усиленный prompt + continuation — чтобы на сканах была хоть какая-то точность. Это для fallback-пути.

**Приоритет:** 🔴 blocker **архитектуры** — без hybrid Recognition нерабочий на реальных ОВиК PDF. Делать сразу после текущей QA-сессии.

**Файлы:**

### #3.1 🔴 Расширение — специфика ОВиК-документации, обнаруженная при углублённом анализе golden PDF

**Вопрос Андрея 2026-04-21:**
> «...в одной строке (типа Название) — быть прописаны (хороший пример на 1 странице "Воздуховод") а на последующих строках НЕ прописаны (пусто), но подразумевается, раз следующий столбец заполнен (в этом примере размерами воздуховодов) то и в пустых строках подразумевается "Воздуховод"...»

**Поправка Андрея (2026-04-21, позже):**
> «...повёрнутый на 90° текст на Спецификациях — это обычно сервисный текст (кто проверил, дата документа и так далее). Такой текст нужно просто игнорировать — он нам совершенно не нужен.»

**Корректная картина после перепроверки:**

**(1) Rotation в метаданных — `page.rotation=90`.**
- Физически PDF portrait (`mediabox 1191×842` уже после rotation), но viewer разворачивает в landscape автоматически.
- После применения `page.set_rotation(0)` или matrix transform bbox'ы становятся стандартными для landscape.
- **Основной контент — `dir=(0, -1)` (104-124 строки на страницу)** — это нормальный горизонтальный текст спецификации после учёта rotation=90.
- **Сервисный штамп — `dir=(-1, 0)`, всего 3 строки** на страницу: «Взам. инв. №», «Подп. и дата», «Инв. № подл.».

**(2) Парсер должен нормализовать rotation и фильтровать сервисный штамп:**
- `page.set_rotation(0)` перед парсингом → весь основной текст становится `dir=(1, 0)`, bbox'ы в стандартных landscape-координатах.
- Фильтр сервисного штампа (любое из условий):
  - `dir` после нормализации отличается от main (`(1,0)`) → сервис.
  - Слова-маркеры в тексте: «Инв. №», «Подп. и дата», «Взам.», «Изм.», «Разраб.», «Пров.», «№ док», «Формат А», «Масштаб», «Лист» (как заголовок столбца, не часть данных).
  - bbox в традиционных зонах штампа: нижняя 10-20% листа + правая 10-15% (где стандартный ГОСТ-штамп ЕСКД).

**(3) Sticky-name внутри страницы + через границы страниц (важнее всего).**
- На p3 **ни в одном блоке нет слова «Воздуховод»** — есть только размеры/ед./кол-во.
- «Воздуховод» введён как заголовок-группа ранее — на p2 (или в начале таблицы на p1/p2).
- **Без cross-page sticky-name все позиции p3 получат пустой `name` = потеряют смысл.**

**(4) Группа-заголовок → дочерние позиции с только размером.**
- Типовой ОВиК-паттерн: в строке «Наименование» один раз пишется «Воздуховод прямоугольный оц.», далее подряд строки с пустым именем и заполненным только размером.
- Item-name должно быть = `{group_header} {specific_size}` (конкатенация). Или: `name = group_header`, а `tech_specs.dimensions = specific_size` — выбрать экспериментально на метрике recall.

**(5) Column-detection через заголовки шапки таблицы.**
- Блоки с текстом «Поз.», «Наименование и техническая характеристика», «Тип, марка, обозначение», «Ед. изме-ре-ния», «Коли-чество» — заголовки колонок таблицы.
- Их bbox (X-координаты в landscape после rotation=0) дают X-диапазоны для извлечения полей из каждой строки-позиции.

**Acceptance для E15.03 на golden:**
- `pages_processed == 9` (все страницы).
- `len(items) >= 130` (≥85% от 152).
- Все items имеют непустой `name` (sticky-name отработал).
- Количество уникальных `section_name` ≥ 5 (в PDF 9 подгрупп: жилая общеобменная, МОП, воздуховоды+изоляция, противодымная-вентиляторы, противодымная-подпор, клапаны типовые, клапаны OKL, фасонные ПДВ, воздуховоды по сечениям).
- Сервисный штамп НЕ попадает в items (≤0 item'ов с текстами «Инв.», «Подп.», «Изм.» и т.п.).
- Нет дубликатов.

**Что передать Пете** (если ещё не начал имплементацию, иначе — в комментарий к PR):
1. rotation нормализация через `page.set_rotation(0)` — делает координаты тривиальными для работы.
2. фильтр сервисного штампа — по dir + keywords + bbox (тройная проверка).
3. sticky-name — внутри и через страницы (критичнее всего).
4. group-header конкатенация.
5. column detection по заголовкам шапки.

**Ошибка ранней версии этой записи** (зачёркнуто для истории): упоминание «row/column inversion» и «таблица напечатана перпендикулярно» — это было неверной интерпретацией `dir=(0,-1)` без учёта `page.rotation=90`. После rotation-нормализации таблица обычная landscape, никакой инверсии координат не нужно.

### #14 🟠 major — sticky parent не работает на артикульных кодах «РЭД-ВВШ-SP-…» (стр 2, строки 15-20)

**Симптом:** в PDF:
- Строка: name=«Выбросной колпак», model=(пусто), остальное пусто.
- Следующие 6 строк: name=(пусто), model=«РЭД-ВВШ-SP-1000х550-10», …/900х250-7, …/800х550-10 и т.д.

Парсер дал:
- 15: name=«Выбросной колпак», model=«РЭД-ВВШ-SP-1000х550-10» ✓ (multi-line склеилась правильно)
- 16-20: name=«РЭД-ВВШ-SP-900х250-7», «…/800х550-10», … — **код стал именем**, модель пустая ✗

**Корневая причина:** после создания 15 sticky=«Выбросной колпак» ✓. Для 16-й позиции buffer=[«РЭД-ВВШ-SP-900х250-7»] (1 строка). В `_split_name_model` вызывается `is_variant_only_line("РЭД-ВВШ-SP-900х250-7")`:
```python
_VARIANT_RE = r"^(?:[фØ]\s*\d+|\d+(?:[.,]\d+)?\s*[xхXХ×]\s*\d+(?:[.,]\d+)?)"
```
Код начинается с «РЭД-», не с `ф/Ø/цифры` → **regex не матчится**, `is_variant_only_line` возвращает `False`. Парсер считает это «обычное имя» → `name=код, model=""`, sticky обновляется в `code` (но sticky нас уже не интересует — нужен был для этой позиции).

**Решение:** расширить `_VARIANT_RE` на артикульные паттерны:
- `[A-ZА-Я0-9]{2,}(?:-[A-ZА-Я0-9]+){2,}` — «РЭД-ВВШ-SP-1000х550-10» (code-like: капслок + цифры + тире, минимум 2-3 сегмента).
- Дополнительно: содержит `\d+[xхXХ×]\d+` в середине — сильный сигнал артикула.
- Исключить: строки с русскими словами-кириллическими словоформами («Клапан», «Воздуховод») — это имена, не артикулы.

**Severity:** major (потеря 5 имён в одном блоке; повторяется в нескольких блоках спеки).

**Root cause:** R4 (variant regex слишком узкий).

### #25 🔴 КРИТИЧЕСКИЙ КОНЦЕПТУАЛЬНЫЙ — dedup суммирует одинаковые позиции из разных систем

**Симптом:** `_deduplicate` в `spec_parser.py` использует ключ `(name, model, brand)` без `section_name`. Две позиции «Огнезащитная клеящая смесь Kleber / Kleber» из «Общеобменной» и «Противодымной» раздела сливаются в одну с суммой количеств.

**Решение (варианты):**

## Стратегические варианты решения (концептуальный выбор Андрея)

Поскольку R1 (column-aware) — причина ≥60% всех проблем, архитектурно есть **три пути**:

### Вариант А — Pure text-layer + column detection (2-3 дня IS-Петя)
- `get_text("dict")` → y-buckets = rows таблицы.
- Определение колонок по column headers («Наименование», «Тип, марка», «Ед. изме-ре-ния», «Коли-чест-во», «Примечание») + их bbox.
- Каждая row парсится по x-координатам: текст в x-range колонки = значение этой колонки.
- Multi-line cells (перенос) — все строки с тем же x-range в ОДНУ row (но разные row-y).
- **Плюс:** 0 LLM вызовов, по-прежнему ≈0.1 с/стр.
- **Минус:** эвристика таблиц — сложна, не покроет все вариации ЕСКД/ГОСТ. Ожидание recall 85-90% на реальных PDF.

### Вариант B — Text-layer + LLM normalization (компромисс скорость/качество) ⭐️
- Text-layer достаёт по bbox rows + column values (как в А).
- **После** text-layer extraction — передать rows списком в LLM (`gpt-4o-mini`) с промптом:
  > «Вот строки таблицы спецификации ОВиК [rows в JSON]. Определи: секционные заголовки, склей переносы name, убери служебные строки штампа, найди колонку Примечание. Верни JSON массив позиций.»
- LLM работает с **текстом**, не с изображением → быстрее и точнее чем Vision (~2-4 с/стр вместо 10).
- 1 LLM call на страницу или 1 на весь документ.
- **Плюс:** recall 92-97% ожидаемый, легко корректируется промптом под новые паттерны.
- **Минус:** LLM-зависимость, $0.001-0.01/документ, ~20-40 с на 19-страничный PDF.
- **Зона компромисса:** Андрей хочет скорость + качество. LLM на готовом тексте — это «быстрая» LLM (работает по тексту, не картинке). Устроит ≈30 с на 20-страничный PDF?

### Вариант C — Специализированная PDF-таблица библиотека (camelot / pdfplumber / tabula)
- Готовые solution'ы для table extraction из PDF.
- Требуют стабильности вёрстки и качество PDF.
- **Плюс:** проверенные алгоритмы.
- **Минус:** плохо работают с ЕСКД-таблицами (сетка часто отсутствует), требуют подбора параметров под каждый тип PDF. Локально 3-5 дней интеграции + валидации.

---

## Моя рекомендация — Вариант B

**Почему:**
1. Андрей явно сказал «разумный компромисс между скоростью и LLM» и «высокая точность». Pure text-layer (A) — упираемся в потолок 85%, проверка вручную = не вариант. Pure Vision — медленно и дорого. **B — именно компромисс.**
2. Text-layer даёт LLM **чистую структуру данных** (bbox, rows, columns) — LLM делает только NLP-работу (нормализация имён, определение секций, фильтрация мусора). Это задача, на которой gpt-4o-mini работает отлично.
3. Экономика: $0.005 на 20-страничный документ × 100 документов/день = $0.5/день. Ничтожно.
4. Скорость: можно делать batch-call (все pages → один запрос) — один roundtrip 10-15 с.

**Риски:** LLM может галлюцинировать позиции. Mitigation — строгий промпт «не добавлять/не изобретать, только структурировать». Regression-тесты на golden.
