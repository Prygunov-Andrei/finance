# Postmortem template

Шаблон для заполнения после каждого P0/P1 инцидента.

Blameless principle: мы не ищем виноватых, мы ищем системные причины и способы предотвращения.

Сохраняется в `docs/incidents/YYYY-MM-DD-short-slug.md`.

---

# Postmortem: [Название инцидента]

**Дата инцидента:** YYYY-MM-DD HH:MM UTC
**Длительность:** XX минут / часов
**Severity:** P0 / P1
**Составил:** [имя], [роль]
**Review:** [имя техлида]

## 1. TL;DR

Одно-два предложения: что случилось, что пострадало, что сделали.

Пример: «15 апреля в 14:23 рухнул Postgres из-за переполнения диска. Production ISMeta был недоступен 42 минуты. Освободили место, восстановили сервис, настроили alert на заполнение диска.»

## 2. Impact

- **Затронутые пользователи:** кто / сколько.
- **Потерянные данные:** что / сколько.
- **SLO impact:** сколько error budget потрачено.
- **Финансовый impact:** если применимо (например, LLM overrun).

## 3. Timeline

Все время в UTC.

| Время | Событие | Кто |
|---|---|---|
| HH:MM | Первый сигнал (alert / user report) | ... |
| HH:MM | Incident open, начало расследования | ... |
| HH:MM | Определена причина X | ... |
| HH:MM | Применён workaround Y | ... |
| HH:MM | Root cause identified | ... |
| HH:MM | Fix applied | ... |
| HH:MM | Service restored | ... |
| HH:MM | Incident closed | ... |

## 4. Root cause

Что **реально** случилось, на глубоком уровне. Не «диск заполнился», а «не было alert'а на заполнение, + cron-job для очистки старых логов отключился 3 месяца назад при миграции, + retention policy не пересматривалась».

## 5. Что сработало хорошо

- Первая реакция была быстрой.
- Runbook помог.
- Backup сработал как ожидалось.
- И т.п.

## 6. Что пошло плохо

- Alert пришёл с задержкой 15 минут.
- Runbook был устаревшим.
- Не было заранее заготовленного скрипта для X.
- Коммуникация с клиентами была хаотичной.
- И т.п.

## 7. Где повезло

- Это произошло днём, не ночью.
- Затронуло только dev-окружение (или только один клиент).
- Было под рукой нужного человека.
- И т.п.

(Важно фиксировать — чтобы понимать, в следующий раз везение может не сработать.)

## Что НЕ включаем

- Имена «виновных».
- Субъективную оценку «кто должен был заметить».
- Наказания.

## Distribution

Postmortem публикуется:
- В `docs/incidents/`.
- В Slack / Telegram команды.
- При security-инциденте — в `#ismeta-alerts` с резюме.

## Follow-up

- Action items трекаются до closure.
- Через 1 месяц — review: помогли ли изменения?
- В monthly review упоминаем закрытые incidents.
