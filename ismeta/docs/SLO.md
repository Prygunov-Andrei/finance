# Service Level Objectives (SLO)

**Версия:** 0.1. **Дата:** 2026-04-15.

Цели доступности, производительности, качества ISMeta. Обновляются после первого месяца эксплуатации в staging/production.

## 1. SLO = Service Level Objective

Для каждого ключевого компонента — измеримая цель. Если долго не дотягиваем — фичи на паузу, фокус на стабильности (error budget policy).

## 2. Доступность

| Компонент | Цель | Допустимый даунтайм/мес | Измерение |
|---|---|---|---|
| Публичный API ISMeta (`/api/v1/*`) | 99.5% | 3ч 39м | synthetic health-check каждую минуту |
| ISMeta Frontend | 99.5% | 3ч 39м | synthetic check |
| Webhook receiver ISMeta | 99.9% | 43м | event processing lag ≤ 60 сек |
| SSE чат агента | 99.0% | 7ч 18м | соединение живёт без ошибок |
| Публичный режим (posle этапа 2) | 99.0% | 7ч 18м | |

**Разъяснение:** 99.5% ≠ "ISMeta работает 99.5% дней". 99.5% = 99.5% времени каждые 30 дней. Даунтайм в 3ч 39м можно распределить как угодно (краткие инциденты или один длинный).

## 3. Латентность

| Операция | Цель | P50 | P95 | P99 |
|---|---|---|---|---|
| GET `/estimates/{id}` | быстро | <100 мс | <300 мс | <800 мс |
| POST/PATCH мутации | быстро | <200 мс | <500 мс | <1500 мс |
| POST `/match-works` (запуск Pass 1) | старт мгновенный | <500 мс | <1000 мс | <2000 мс |
| Pass 1 завершение (смета 4000 строк) | приемлемо | <60с | <90с | <120с |
| Pass 2 завершение (с LLM) | терпимо | <8мин | <15мин | <20мин |
| Агент — ответ без tool | быстро | <3с | <8с | <15с |
| Агент — ответ с 1 tool-вызовом | приемлемо | <6с | <15с | <30с |
| Экспорт PDF | приемлемо | <5с | <15с | <30с |
| Экспорт XLSX | быстро | <2с | <5с | <10с |
| Webhook обработка | быстро | <500 мс | <2с | <5с |

## 4. Качество подбора (бизнес-SLO)

Регрессионные пороги на golden set (фиксируются после первого прогона, см. [`specs/07-mvp-acceptance.md §3.2`](../specs/07-mvp-acceptance.md)).

| Метрика | Минимум | Целевое |
|---|---|---|
| Precision (матчинг работ) | baseline - 5% | baseline + 10% |
| Recall | baseline - 5% | baseline + 10% |
| Стоимость LLM на среднюю смету | < 150 ₽ | < 50 ₽ |
| Доля смет без unmatched позиций | > 70% | > 90% |

## 5. Error budget

### 5.1 Определение

Error budget = `(1 - SLO) * период`.

Для «99.5% доступности в месяц» = 0.5% * 30 дней = 3 часа 39 минут даунтайма в месяц.

### 5.2 Policy

- **Бюджет не потрачен (< 50%):** команда фокусируется на фичах.
- **Бюджет потрачен на 50-80%:** повышенное внимание к стабильности, code review строже.
- **Бюджет потрачен на 80-100%:** новые фичи в PR требуют обоснования «не влияет на стабильность», приоритет — postmortem и исправление уже случившихся инцидентов.
- **Бюджет превышен:** заморозка фич на месяц, все руки — на стабильность.

## 6. Error budget по трём месяцам MVP (адаптация)

Первые 3 месяца после запуска — особый режим:

- SLO смягчаются до 98% (значит error budget 14 часов в месяц);
- каждый инцидент обязателен postmortem с обновлением runbook'а;
- после 3 месяцев — целевые SLO из п.2.

## 7. Измерение и отчётность

### 7.1 Сбор данных

- **Синтетические health-check'и** — external service (Uptime Robot / Healthchecks.io / self-hosted Blackbox Exporter).
- **Real User Monitoring** — для latency frontend (Sentry Performance или отдельный).
- **Application metrics** — из `MetricAggregate` (см. [`specs/11-metrics.md`](../specs/11-metrics.md)).

### 7.2 Дашборд SLO

`/admin/slo/` — отдельная админская страница:
- текущий uptime за месяц (процент и остаток error budget);
- latency P50/P95/P99 по ключевым операциям;
- график выгорания budget.

### 7.3 Отчёт

Ежемесячный отчёт техлида команде: какие SLO достигнуты, где нет, что делаем.

## 8. Cache staleness SLO

Для кеша Product (см. [ADR-0006](./adr/0006-no-workitem-cache.md) + stale-while-revalidate):

| Метрика | Цель |
|---|---|
| P50 staleness (время от изменения Product до применения в кеше) | < 30 сек (webhook) |
| P95 staleness | < 15 мин (TTL) |
| P99 staleness | < 60 мин (polling fallback при потере webhook'ов) |
| Max acceptable staleness | 4 часа (дальше — блок мутаций, warning в UI) |
