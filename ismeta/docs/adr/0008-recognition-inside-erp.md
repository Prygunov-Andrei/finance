# ADR-0008. Сервис распознавания — отдельное Django-приложение внутри ERP

- **Статус:** Accepted
- **Дата:** 2026-04-15

## Контекст

Распознавание PDF/Excel → структурированный JSON. Варианты размещения:
1. **Внутри ISMeta** — часть ISMeta-backend'а.
2. **Отдельное Django-приложение в ERP** — `backend/recognition/`.
3. **Полностью отдельный микросервис** на своём сервере.

В текущем ERP есть `backend/llm_services/services/specification_parser.py` — прототип распознавания.

## Решение

Вариант 2: отдельное Django-приложение внутри ERP с публичным HTTP API.

## Обоснование

- **Не в ISMeta:** распознавание нужно и публичному сайту, и будущим коробочным клиентам, не только сметчику. Делать часть ISMeta = замусорить ISMeta.
- **Не микросервис:** распознавание нужен доступ к каталогу ERP (нормализация имён), к LLM-шлюзу, к storage. Отдельный сервер = дублирование. Оверинжиниринг для одного клиента (наш Август).
- **Django-приложение в ERP:** переиспользует инфраструктуру ERP (auth, storage, LLM-шлюз), но имеет свой контракт наружу. ISMeta вызывает через HTTP, не через импорт Python-модулей.
- **Коробочный сценарий:** приложение `recognition/` идёт в бандле коробки — работает автономно внутри той же Django-инсталляции.

## Последствия

- **Плюс:** минимальная сложность, переиспользование инфраструктуры.
- **Минус:** формально привязаны к Django. Но мы и так на Django.
- **Минус:** доступно только при живом ERP. Для публичного режима ISMeta (который тоже живёт внутри ISMeta) — нужна альтернатива: либо дублирование в ISMeta, либо publicMode в ISMeta вызывает recognition ERP через тот же API. Выбрано второе.

## Триггер к пересмотру

Если распознавание нужно масштабировать отдельно (клиент грузит сотни PDF в день, нужно больше GPU/LLM-воркеров) — выделяем в микросервис. Контракт уже определён как HTTP API, переход совместимый.
