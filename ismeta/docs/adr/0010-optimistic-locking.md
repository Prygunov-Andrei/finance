# ADR-0010. Optimistic locking на строках сметы вместо CRDT

- **Статус:** Accepted
- **Дата:** 2026-04-15

## Контекст

Правило «одна смета — один сметчик» — организационное. Технически сметчик может открыть смету в двух вкладках или в виджете + standalone. Нужна защита от потери изменений.

Варианты:
1. **Pessimistic lock:** блокировать строку на время редактирования.
2. **Optimistic lock:** поле `version`, проверка на UPDATE.
3. **CRDT / OT:** real-time collaborative (как Google Docs).

## Решение

Optimistic locking.

## Обоснование

- **CRDT/OT** — избыточно. Сценарий «два сметчика одновременно правят одну смету» для нас редкий.
- **Pessimistic lock** создаёт проблемы с зависшими сессиями (лок не снят, строку никто не трогает, другой сметчик ждёт).
- **Optimistic** — 409 Conflict при рассинхроне, клиент показывает диалог «данные изменились, перезагрузить или перезаписать?». Простой и эффективный.

## Реализация

- Поле `version: INT` на `EstimateItem`, `EstimateSection`, `EstimateSubsection`, `Estimate`.
- На UPDATE — `WHERE id = ? AND version = ?` с инкрементом. Если 0 rows affected — 409.
- Клиент передаёт текущий `version` в заголовке `If-Match`.
- Middleware/serializer обрабатывает 409 унифицированно.

## Последствия

- **Плюс:** простая реализация, предсказуемое поведение.
- **Минус:** нет real-time-совместного редактирования. Если понадобится — добавим WebSocket-канал в отдельном эпике (этап 3+).
- **Минус:** чуть больше конфликтов 409 в двух-вкладочном сценарии. UI обязан элегантно их обрабатывать.
