# ADR-0003. Multi-tenancy через UUID `workspace_id` с первого дня

- **Статус:** Accepted
- **Дата:** 2026-04-15

## Контекст

ISMeta задумывается как продукт, пригодный для:
- коробочных клиентов (один Workspace на инсталляцию);
- публичного режима Августа (отдельный Workspace для посетителей);
- будущего SaaS (много Workspace в одной БД).

Можно не закладывать multi-tenancy сразу («в коробке всё равно один клиент»). Потом пожалеем.

## Решение

Каждая таблица с данными имеет `workspace_id UUID NOT NULL`. Даже если в MVP Workspace всегда один.

## Обоснование

- **UUID, не INT:** если завтра сольём два коробочных инстанса в SaaS — UUID не конфликтуют, INT конфликтуют.
- **С первого дня:** добавление `workspace_id` в работающую БД с сотнями миллионов строк — адский труд, с миграцией всех внешних FK. На старте это копеечная графа.
- **FilterBackend:** DRF-слой обязан фильтровать по `workspace_id`. Тесты проверяют изоляцию.
- **Celery:** задачи помечаются `workspace_id`, очереди могут быть per-workspace при нагрузке.

## Последствия

- **Плюс:** SaaS включается без переписывания. Early-test mode с двумя Workspace выявляет баги изоляции сразу.
- **Минус:** каждый запрос содержит `WHERE workspace_id = ?`. Это и в индексах, и в join'ах. Небольшая накладная.
- **Минус:** сложнее отладка — всегда надо помнить о workspace context.

## Проверка

- Все тесты имеют фикстуру с двумя `workspace_id`, проверяют что запросы одного не возвращают данные другого.
- Каждая миграция, добавляющая таблицу, включает `workspace_id`.
- Middleware `WorkspaceMiddleware` кладёт `workspace_id` в `request` из JWT.
