# ADR-0005. LLM-агент через tool use, а не через полный JSON-контекст

- **Статус:** Accepted
- **Дата:** 2026-04-15

## Контекст

LLM-агент помогает сметчику. Два подхода:
1. **Полный контекст:** каждый запрос к LLM включает JSON всей сметы (2-4 тыс. строк).
2. **Tool use:** LLM имеет набор инструментов (`get_item`, `find_alternatives_by_specs`), вызывает их по необходимости.

## Решение

Tool use (ReAct-style).

## Обоснование

- **Цена токенов:** смета в 2 тыс. строк — ~60K токенов. Каждый вызов gpt-4o — $0.30+. При 10 сообщениях чата — $3+ за одну смету. Неприемлемо.
- **Свежесть данных:** между сообщениями сметчик может править строки. Полный контекст протухает.
- **Локальные LLM:** контекст-окна у Ollama 4-32K. Полная смета не влезает.
- **Прозрачность:** по логам видно, какие tool'ы вызвал LLM — легче отлаживать.

## Последствия

- **Плюс:** дёшево, точно, работает с разными провайдерами.
- **Минус:** выше latency — каждый tool-вызов — это дополнительный round-trip.
- **Минус:** надо проектировать набор tool'ов и их сигнатуры. Принципиальный дизайн.

## Альтернативы, которые рассматривали

- **RAG-like approach:** превратить смету в векторную БД, доставать релевантные строки. Технически сложнее, не даёт ощутимых преимуществ для нашего размера смет.
- **Hybrid:** короткое summary сметы в каждом запросе + tools. Оставляем на будущее, если summary окажется недостаточно.
