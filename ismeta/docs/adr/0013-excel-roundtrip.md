# ADR-0013. Excel-цикл через row_id + hash, без Google Sheets в MVP

- **Статус:** Accepted
- **Дата:** 2026-04-15

## Контекст

Сметчик для массовых правок хочет уходить в привычную среду. Варианты:
1. **Excel-цикл:** выгрузка .xlsx, правка, импорт обратно.
2. **Google Sheets:** OAuth, двусторонняя синхронизация.
3. **Встроенный грид** в UI ISMeta (AG Grid, Handsontable).

## Решение

Excel-цикл через скрытые `row_id` (стабильный UUID) и `row_hash` (хэш значимых полей). Google Sheets — в backlog.

## Обоснование

- **Excel привычнее** — 100% сметчиков его знают.
- **Офлайн:** Excel работает без интернета.
- **Корпоративные политики:** у коробочных клиентов часто нет доступа к Google.
- **Google Sheets сложнее:** OAuth каждого пользователя, rate limits API, conflict resolution, шифрование ПДн в облаке Google. Отложим.
- **Встроенный грид** — хорошее решение, но inline-редактор ISMeta уже достаточен для типовых сценариев; массовые правки — в Excel.

## Реализация (см. [`specs/05-excel-schema.md`](../../specs/05-excel-schema.md))

- При экспорте файл содержит 3 листа: данные, скрытые метаданные, инструкция.
- Скрытые столбцы: `row_id` (стабильный UUID), `row_hash` (sha256 от значимых полей на момент экспорта).
- При импорте:
  - сверка по `row_id`;
  - сравнение `row_hash` на файле с текущим hash в БД — если различается, значит в БД было изменено после экспорта, three-way merge;
  - fallback по `(name, model, unit, quantity)` если `row_id` искажён.

## Последствия

- **Плюс:** удобно сметчику, работает офлайн, ничего не течёт в Google.
- **Минус:** сложный алгоритм импорта (three-way merge). Написание и тестирование — отдельный эпик E7.
- **Минус:** если сметчик сохраняет .csv вместо .xlsx — row_id теряется, нужен fallback-режим.
