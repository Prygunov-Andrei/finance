# ADR-0004. Outbox + polling fallback для webhook'ов от ERP

- **Статус:** Accepted
- **Дата:** 2026-04-15

## Контекст

ERP уведомляет ISMeta об изменениях (товар обновился, договор подписан). Варианты:
1. **Синхронный HTTP-вызов** в конце бизнес-транзакции ERP.
2. **Kafka/RabbitMQ** — отдельная очередь с durability.
3. **Outbox pattern:** события в таблицу ERP, фоновый воркер пушит в ISMeta + polling fallback со стороны ISMeta.

## Решение

Outbox + polling fallback.

## Обоснование

- **Синхронный вызов** — хрупок: если ISMeta недоступна, бизнес-операция в ERP либо падает, либо «теряет» событие.
- **Kafka** — избыток для наших объёмов (десятки событий в секунду максимум). Требует команды для обслуживания.
- **Outbox** — даёт at-least-once гарантию без новой инфраструктуры; polling fallback закрывает случай «воркер умер, webhook'и не отправляются часами».

## Последствия

- **Плюс:** простой стек (только Postgres + Celery), гарантии доставки, асинхронность.
- **Плюс:** легко диагностировать — всё видно в `erp.outbox`.
- **Минус:** задержка. Нормально для нашего домена (данные каталога — не realtime).
- **Минус:** нужно писать в `outbox` в той же транзакции, что и доменное изменение. Дисциплина.

## Возможная эволюция

Если нагрузка вырастет и polling-частота начнёт заметно мешать — перейдём на Kafka. Переход совместимый: события уже структурированы с `event_id` и `schema_version`.
