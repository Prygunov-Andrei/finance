# ADR-0022. Полукоробочный продукт: под Август, с архитектурой для адаптации

- **Статус:** Accepted
- **Дата:** 2026-04-17
- **Источник:** ретроспектива 4 customer development интервью, предложение Андрея (PO).

## Контекст

Исходная позиция CONCEPT.md: «ISMeta — коробочный продукт, первый клиент — Август Климат». Это подразумевало generic UI, generic workflow, много настроек «под любую компанию».

После 4 интервью (3 внутренних + 1 внешний) обнаружено:

1. **Два ICP** с кардинально разными процессами (профсметчик vs инженер-руководитель).
2. **Процесс Августа** — конкретный, устоявшийся, со своими артефактами (форма внутренней сметы, формат ТКП, коэффициенты 3.5/1.3/3.2, 6 связанных документов).
3. **«Коробка для всех»** в MVP = размывание фокуса, более долгая разработка, менее точное попадание в боли конкретного пользователя.
4. Август — dogfood-клиент и единственный клиент MVP. Generic UI на этом этапе — overengineering.

## Решение

**Полукоробочный продукт:**

- **UI и workflow** = конкретный процесс Август Климат (PDF → смета → поставщики → проверка → передача в ERP).
- **Архитектура** = tenant-agnostic, легко адаптируемая.

### Что «под Август» (жёстко в MVP)

- Процесс: получил проект PDF → распознал → внутренняя смета → запросы поставщикам → монтажные цены из прайса → агрегаты → проверка → передача в ERP.
- UI-layout: таблица смет с keyboard-first навигацией.
- Интеграция с ERP Август через webhook/snapshot.

### Что «для адаптации» (заложить с первого дня)

| Что | Где | Зачем |
|---|---|---|
| `workspace_id` на каждой таблице | ADR-0003 | Изоляция данных |
| Коэффициенты (3.5/1.3/3.2) | `WorkspaceSettings` JSONB | Другая компания = другие коэффициенты |
| Монтажный прайс | Отдельная сущность, загружаемая per workspace | Свой прайс у каждого |
| Список поставщиков | Per workspace | Свои поставщики |
| Формат экспорта ТКП | Template per workspace/customer | АСГ хочет разделение монтаж/оборудование |
| Правила подбора работ | ProductKnowledge per workspace + .md файлы | Свои правила для ОВиК vs СС vs общестрой |
| LLM-провайдер | `WorkspaceSettings.llm_provider_id` | Разные клиенты = разные API |
| Feature flags | Per workspace в БД (ADR-0019) | Включать фичи выборочно |

### Граница ISMeta ↔ ERP

ISMeta **отдаёт:**
- Таблицу сметы (все строки, цены, работы).
- Агрегаты: итого, итого по разделам, оборудование/материалы/работы.
- Человеко-часы, прибыльность, коэффициенты.
- Расчёт аванса (формула), расчёт сроков (формула).

ERP **делает сам:**
- ТКП (лист A4 с данными из snapshot сметы).
- Договор, КС-ки, допсоглашения.
- Коммуникация с заказчиком.

ISMeta **не создаёт** ContractEstimate/ContractAmendment в ERP (ADR-0014 остаётся).

## Обоснование

1. **Фокус.** MVP для одного клиента (Август) = быстрее, точнее, можно итерировать по реальной обратной связи. Generic UI на этом этапе — premature abstraction.
2. **Скорость.** Конкретный процесс = конкретные экраны = меньше решений «а как это должно работать для произвольной компании?».
3. **Архитектура не страдает.** `workspace_id`, JSONB settings, per-workspace конфиги — уже заложены. Адаптация под новую компанию = новый workspace + конфиги + прайс. Код не меняется.
4. **Путь к коробке.** Когда ISMeta отработана на Августе → обобщаем UI → добавляем onboarding wizard → коробка для этапа 3.

## Триггер пересмотра

- Появился второй реальный клиент с процессом, **несовместимым** с текущим UI → нужен generic layer.
- Август перестал быть dogfood (ушёл на другой продукт) → нужен новый якорный клиент.
- >3 клиентов просят одну и ту же фичу, которой нет → обобщаем.

## Последствия

### Плюсы

- Быстрее MVP (фокус на одном процессе).
- Точнее попадание в боли (конкретный пользователь = конкретная обратная связь).
- Архитектура уже tenant-agnostic — не создаёт техдолг.

### Минусы

- Второй клиент может потребовать переделки UI (но не архитектуры).
- Нет «демо для произвольной компании» — только демо на процессе Августа.
- Risk: привязка к одному клиенту. Если Август не будет использовать — нет обратной связи.

### Митигации

- Архитектурные конфиги в JSONB — адаптация без изменения кода.
- Customer development с внешними компаниями продолжается (следующий раунд — при наличии прототипа).
- UI компонентный (shadcn/ui) — перекомпоновка экранов дешёвая.
