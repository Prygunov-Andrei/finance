# ADR-0016. Excel round-trip: упрощённый вариант в MVP

- **Статус:** Accepted
- **Дата:** 2026-04-15
- **Источник:** MVP-SIMPLIFICATIONS §1, PO-ревью D2.

## Контекст

В ADR-0013 была принята полная схема three-way merge при Excel-импорте:
- сверка по `row_id` и `row_hash`;
- баcline state для детекции конфликтов;
- UI для resolution conflicts.

Реальная реализация это — 2-3 недели разработки (XL-эпик E7). Непропорциональная ценность для MVP, где:
- Один сметчик работает с одной сметой.
- Concurrent editing — редкое событие.
- Real-world demand для conflict resolution — неизвестна.

## Решение

**MVP-версия Excel round-trip:**
- Экспорт — полный (с скрытыми `row_id` и `row_hash`).
- Импорт — простая сверка по `row_id`:
  - совпадает → обновление;
  - новая строка (нет row_id) → добавление;
  - отсутствует в файле → soft delete (пометка `is_deleted`);
  - `row_id` не найден в БД → warning «возможно, вы импортируете старый файл, продолжить?».
- Hash-сверка: если hash отличается — **«last write wins»** с уведомлением сметчика.
- Diff-preview остаётся (простой: показать, что добавилось/удалилось/изменилось).
- НЕТ three-way merge.

## Триггер пересмотра

Делаем полный three-way merge, если:
- Клиент просит более 3 раз «я правил одновременно с коллегой, нужен conflict resolution».
- DAU > 10 с частыми параллельными правками одной сметы.
- UX-ревью после MVP показывает, что это главный pain point.

## Последствия

### Плюсы

- Эпик E7: XL → M (2 недели → 3 дня).
- Достаточно для сценария «один сметчик, одна смета».
- Код простой и maintainable.

### Минусы

- «Last write wins» — чревато потерей изменений, если реально два parallel edits.
- Будем объяснять сметчику: «не редактируйте одновременно».
- При появлении collab — нужен redo.

### Митигации

- UI подсказка: «Рекомендуем не редактировать смету одновременно в Excel и в ISMeta».
- AuditLog фиксирует hash-конфликты — при частоте > 5/месяц → сигнал пересмотреть.
- Organizational rule: «Одна смета — один сметчик».
