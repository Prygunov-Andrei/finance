# ADR-0002. Отдельная БД PostgreSQL, не общая с ERP

- **Статус:** Accepted
- **Дата:** 2026-04-15

## Контекст

ISMeta и ERP могут:
1. **Делить одну БД Postgres** — через схемы или префиксы таблиц.
2. **Каждый со своей БД** — общение только через API.

Первый вариант соблазнителен («просто join'ы между сметой и договором»).

## Решение

Отдельная БД для ISMeta. Общение с ERP только через REST API + webhooks.

## Обоснование

- **Изоляция развёртывания:** ISMeta можно выкатить на отдельный сервер у коробочного клиента.
- **Независимое версионирование миграций:** ISMeta мигрирует, не опасаясь ломать ERP.
- **Чёткая граница ответственности:** кто владелец данных — определяется по БД. Никаких «ну я тут в чужую таблицу залез».
- **Future SaaS:** в режиме SaaS ISMeta живёт в облачной БД, а ERP клиентов — у них локально. Join'ы невозможны.
- **Безопасность:** компрометация одной БД не даёт доступ к другой.

## Последствия

- **Плюс:** отделённость, переносимость, безопасность.
- **Минус:** дубликация кеша справочников (каталог, прайсы) в ISMeta. Решается через `ProductCache` + webhooks.
- **Минус:** нельзя делать join в SQL между `EstimateItem` и `Contract`. Решается API-запросами с обогащением.
