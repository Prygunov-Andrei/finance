# MVP-упрощения: что сознательно НЕ делаем в этапе 1

**Дата:** 2026-04-15. **Версия:** 0.1. **Источник:** категория Б в REVIEWS-RECONCILIATION.md.

Сводка из 20 сознательных решений «отказать / отложить / упростить» в MVP. Каждое — чтобы сфокусировать ресурсы на критических фичах и избежать overengineering.

## Принцип

> Максимум из минимума. Лучше 5 отличных фич, чем 15 средних.

Каждое упрощение — не «не будем никогда», а «не в MVP». С триггером пересмотра.

---

## 1. Excel round-trip — упрощённый вариант

**Было:** полный three-way merge с детекцией конфликтов по row_id + row_hash + baseline.

**Упрощение в MVP:**
- экспорт с row_id в скрытом столбце — есть.
- импорт обратно — сверка только по row_id.
- при несовпадении row_id (сметчик удалил скрытый столбец) — показываем warning «импорт заблокирован, экспортируйте заново».
- конфликты (строка правилась и в БД, и в файле) — последнее wins, с уведомлением сметчика.

**Триггер пересмотра:** клиент или сметчик просит три-way merge более 3 раз.

**См. ADR-0016.**

---

## 2. Публичный режим — в этап 2.5

**Было:** публичный режим — часть этапа 2.

**Упрощение:**
- Этап 2: только полный LLM-агент + metrics dashboard + ProductKnowledge review.
- Этап 2.5: публичный режим (после закрытия первого внутреннего пользователя).

**Почему:** публичный режим отвлекает ресурсы от полировки MVP. Без эксплуатации MVP не понимаем, что важнее.

**Триггер пересмотра:** готовность MVP внутреннего пользователя + 2 недели стабильной работы.

---

## 3. LLM-агент — только 2 инструмента в MVP

**Было:** 7 инструментов в этапе 2.

**Упрощение (уже принято в specs/04):**
- `get_item(item_id)` — техническая основа.
- `find_alternatives_by_specs(item_id)` — главная ценность.
- Остальные (`get_estimate_summary`, `get_section`, `search_knowledge`, `get_supplier_prices`, `propose_change`, `apply_change`) — этап 2+.

**Триггер пересмотра:** сметчик в dogfood-режиме просит больше функций более 5 раз.

---

## 4. EstimateSubsection — за feature flag

**Было:** полноценная поддержка 3-уровневой иерархии.

**Упрощение (уже принято):**
- Модель и миграция — есть.
- UI скрыт за `Workspace.settings.feature_flags.subsection_ui_visible = false`.
- Через 6 месяцев эксплуатации — решаем: включаем или удаляем.

**Триггер пересмотра:** 6 месяцев после MVP.

---

## 5. Multi-ERP на один Workspace — не поддерживается

**Было:** задать вопрос «что если холдинг с 5 юр.лицами и 5 ERP?»

**Упрощение:**
- Правило: один Workspace = один ERP.
- Холдинг = несколько Workspace.
- Модель поддерживает архитектурно (UUID workspace_id), но API не рассчитан на cross-workspace queries.

**Триггер пересмотра:** первый холдинг-клиент.

---

## 6. Event sourcing — отложено

**Было:** параллельная shadow-таблица `domain_event`.

**Упрощение:**
- В MVP — только CRUD + AuditLog.
- Event sourcing — backlog (если потребуется replay для compliance).

**Триггер пересмотра:** compliance-требование от клиента или запрос «покажи все действия по смете с возможностью отмены любого».

---

## 7. CQRS — отложено

**Было:** read-replica для отчётов / дашборда.

**Упрощение:**
- В MVP — всё из primary БД.
- Read-replica добавляем по триггеру.

**Триггер пересмотра:** p95 latency на `/admin/metrics` > 2 секунд.

---

## 8. Redis HA — отложено

**Было:** Redis Sentinel / Cluster с failover.

**Упрощение:**
- Один Redis инстанс.
- Graceful degradation при падении (нет webhook idempotency → повторы ошибок; нет Celery → задержка подборов).
- Backup Redis через dump.rdb раз в сутки.

**Триггер пересмотра:** > 10 сметчиков в production или первый production-инцидент с Redis.

---

## 9. SSE не выносим в отдельный сервис

**Было:** отдельный Node.js / Go для SSE чата.

**Упрощение:**
- Django ASGI + daphne для SSE.
- Лимит concurrent chat sessions: 50.

**Триггер пересмотра:** > 30 concurrent chat sessions или signifcant chat latency.

---

## 10. Recognition внутри ERP (не микросервис)

**Было:** отдельный сервис для масштабирования распознавания.

**Упрощение (ADR-0008):**
- Django-app `recognition/` внутри ERP.
- Разделяет инфраструктуру с ERP.

**Триггер пересмотра:** > 100 PDF/день или реальная ботleneck.

---

## 11. Single region infrastructure

**Было:** multi-region для disaster recovery.

**Упрощение:**
- Один сервер (216.57.110.41) для нашего инстанса.
- Backup в другой регион S3 (Yandex Cloud).
- RTO 2 часа, RPO 15 минут.

**Триггер пересмотра:** первый корпоративный клиент с DR-требованиями (SLA uptime > 99.9%).

**См. ADR-0017.**

---

## 12. Docker Compose вместо Kubernetes

**Было:** k8s для переносимости и scaling.

**Упрощение:**
- Docker Compose — для коробочных клиентов.
- Docker Compose — для нашего production (в этапе 1-2).
- Kubernetes — когда начнём SaaS (этап 4+).

**Триггер пересмотра:** запуск SaaS или > 20 коробочных клиентов.

**См. ADR-0018.**

---

## 13. Feature flags в БД без UI

**Было:** отдельная админка для флагов.

**Упрощение:**
- Таблица `FeatureFlag` в БД.
- Переключение через Django admin (стандартный).
- Без A/B testing, без gradual rollout в MVP.

**Триггер пересмотра:** > 10 активных флагов или необходимость gradual rollout.

**См. ADR-0019.**

---

## 14. Bug bounty — не в MVP

**Было:** программа bug bounty для публичного режима.

**Упрощение:**
- Private responsible disclosure (адрес `security@august.ru`).
- Bug bounty — через 12+ месяцев после production launch.

**Триггер пересмотра:** > 3 reported vulnerabilities через private disclosure.

---

## 15. Только desktop UI

**Было:** рассмотреть мобильный доступ.

**Упрощение (уже принято):**
- Desktop-only для сметчика.
- Публичный режим — responsive (для посетителей с мобильных).
- Коробочный клиент может использовать с планшета (1024px+) — работает, но не оптимизировано.

**Триггер пересмотра:** запрос клиента на мобильный UX (не просмотр, а редактирование).

---

## 16. Тёмная тема — опциональна

**Было:** тёмная тема обязательна (8 часов с цифрами).

**Упрощение:**
- Light theme по умолчанию.
- Dark theme — toggle в настройках пользователя.
- Sparingly customizable (не делаем редактор цветов).

**Триггер пересмотра:** просьба сделать тёмную тему default от 50%+ сметчиков.

---

## 17. LLM-агент — короткая память

**Было:** long-term memory («сметчик не любит Vaillant»).

**Упрощение:**
- Context per смета (сохраняется пока открыта смета).
- Short-term history — последние 20 сообщений в рамках chat session.
- Long-term preferences — backlog.

**Триггер пересмотра:** явный запрос сметчика.

---

## 18. Self-serve onboarding — отложено

**Было:** «клиент разворачивает сам за час».

**Упрощение:**
- В MVP — мы едем к клиенту, устанавливаем руками.
- install.sh есть, но использует его наша команда.
- Self-serve — этап 4+ (SaaS).

**Триггер пересмотра:** > 5 коробочных клиентов, невозможно вручную.

---

## 19. Интеграции с 1С/CRM — отложены

**Было:** обязательны для российского рынка.

**Упрощение:**
- В MVP ISMeta работает с ERP Август (не 1С).
- Интеграции с 1С, Битрикс24, AmoCRM — backlog.

**Триггер пересмотра:** первый клиент, просящий 1С-интеграцию.

---

## 20. Админ-UI через Django admin

**Было:** собственный Next.js admin panel с богатым UX.

**Упрощение:**
- Django admin (бесплатно, некрасиво, работает).
- В нём: управление Workspace, Members, FeatureFlags, LLMBudgets.
- Кастомная часть в Next.js — только metrics dashboard.

**Триггер пересмотра:** админы жалуются на Django admin UX более 3 раз.

**См. ADR-0020.**

---

## Что ЯВНО остаётся в MVP

Чтобы не было сомнений, вот что **делаем в MVP** (для балансирования этого документа):

- Полная модель Estimate + версии + optimistic locking.
- Markup системы 3 уровня.
- 8-уровневый pipeline подбора работ.
- LLM-агент с 2 инструментами.
- Excel экспорт + простой импорт.
- Distributed tracing (OpenTelemetry).
- Three-level health checks.
- Basic metrics dashboard.
- Audit log (1 год).
- Partitioning критичных таблиц.
- Multi-tenancy через workspace_id.
- LLM fallback chain.
- Stale-while-revalidate для Product cache.
- HMAC для webhooks.
- JWT auth с refresh.
- VersionLink для ISMeta ↔ ERP.

## Регулярный ретроспектив упрощений

Каждые 3 месяца на monthly review — проходимся по этому списку:
- Какой из триггеров сработал?
- Нужно ли ре-анализировать?
- Появились ли новые кандидаты на упрощение?

Этот документ — **живой**. Упрощения обновляются, новые — добавляются.
