# Сотрудничество руководителя с Claude (архитектурный консультант)

**Версия:** 0.1. **Дата:** 2026-04-15.

Этот документ описывает, как работать в паре: руководитель проекта (PO/бизнес) + Claude (техлид-консультант). Для первого проекта — это ваш основной инструмент управления.

## 0. Контекст

- Руководитель — Андрей, впервые ведёт проект разработки.
- Claude — синтетический техлид/архитектор/ревьюер, доступный по запросу.
- Команда разработчиков — будет собрана, но пока не готова (TEAM.md = TBD).
- Продукт — ISMeta, на этапе «архитектура готова, код не начат».

## 1. Распределение ролей

### 1.1 Руководитель делает

- **Принимает финальные решения** (архитектурные, бизнесовые, приоритетные).
- **Взаимодействует с людьми** (найм, переговоры с ERP-командой, внешние контакты).
- **Ведёт коммерческую часть** (клиенты, юристы, бюджет).
- **Отвечает на вопросы Claude** для прояснения неопределённостей.
- **Валидирует предложения** Claude до принятия.
- **Фиксирует итоги** в документах (через PR или прямое обновление).

### 1.2 Claude делает

- **Пишет и поддерживает документы** (архитектура, спеки, ADR, ревью).
- **Отвечает на технические вопросы** в широкой перспективе.
- **Ищет и находит** информацию в репо, обновляет.
- **Проводит ревью** (архитектурные, security, UX, DevOps, продуктовые).
- **Планирует декомпозицию** задач и эпиков.
- **Объясняет технические решения** понятно для нетехнического контекста.
- **Делает черновую работу** (написать план, собрать данные, проанализировать).

### 1.3 Что НИКТО из нас не делает

- **Claude не пишет production-код.** Только спецификации, примеры, skeletons. Реальный код — разработчики команды.
- **Claude не взаимодействует с внешними людьми.** Не пишет письма клиентам, не ведёт переговоры.
- **Claude не хранит состояние между сессиями.** Вся память — в git-репозитории. Если решение не задокументировано — оно потеряется.
- **Руководитель не пишет документацию сам.** Это работа Claude.

## 2. Ритм работы

### 2.1 Синхронные сессии (руководитель + Claude одновременно)

| Ритм | Цель | Длительность |
|---|---|---|
| **Daily standup** (по желанию) | Статус прогресса, быстрые вопросы | 15-30 мин |
| **Weekly review** (обязательно) | Прогресс недели, планы на следующую, открытые вопросы | 1-2 часа |
| **Monthly review** | Ретроспектива, корректировка roadmap, новые ревью | 3-4 часа |
| **Ad-hoc session** | По запросу — обдумать конкретную проблему | 30 мин - 3 часа |

### 2.2 Асинхронные действия

- Claude между сессиями — не «живёт» (нет фонового процесса).
- Руководитель между сессиями — читает документы, общается с командой, принимает бизнес-решения.
- Следующая сессия — с обновлённым контекстом (через git-состояние репо).

### 2.3 Что приносить на каждую сессию

**На weekly review:**
- Что сделано за неделю (команда + внешние действия).
- Проблемы / блокеры.
- Предстоящие решения.
- Вопросы к Claude.

**На monthly review:**
- Roadmap progress (% по фазам reconciliation).
- Метрики (если есть).
- Изменения рынка / конкурентов.
- Новое ревью (если назрело).

## 3. Как задавать вопросы Claude

### 3.1 Хороший запрос

- **Контекст:** что мы уже обсуждали (через ссылку на документ).
- **Конкретная проблема:** что именно неясно.
- **Вариант(ы) решения:** если есть собственные идеи.
- **Желаемый формат ответа:** ревью / список вариантов / конкретное решение / документ.

**Пример:**
> «В specs/04-llm-agent.md мы приняли 2 инструмента в MVP. Команда спрашивает: что если сметчик захочет сравнить несколько аналогов одновременно? Нужно ли добавить `compare_alternatives` или расширить `find_alternatives_by_specs`? Дай 3 варианта с pros/cons.»

### 3.2 Плохой запрос

- «Что делать дальше?» (без контекста).
- «Сделай хорошо.» (без критериев).
- «Посмотри весь проект и скажи.» (слишком широко).

Claude в такой ситуации задаст уточняющие вопросы — это трата времени сессии.

## 4. Как принимать решения

### 4.1 Процесс для архитектурных решений

1. Возникла развилка / вопрос.
2. Claude готовит **2-3 варианта** с pros/cons.
3. Руководитель принимает решение (или просит больше вариантов).
4. Claude оформляет **ADR** в `docs/adr/NNNN-...md`.
5. Если решение влияет на спеки — обновляет их.

### 4.2 Процесс для бизнес-решений

1. Возникает бизнес-вопрос (цена, клиент, юрист).
2. Claude даёт **контекст и анализ**, НЕ принимает решение.
3. Руководитель принимает решение (возможно после консультации с внешними).
4. Фиксируется в соответствующем документе (PRICING.md, GTM.md, etc.).

### 4.3 Процесс для технических изменений

1. Разработчик делает PR.
2. Ревью через GitHub (не через Claude — он не видит PR в реальном времени).
3. Claude можно попросить: «прочитай diff этого PR и оцени» — даст свой взгляд.
4. Merge — решение разработчика/техлида-человека.

## 5. Log решений

Каждое принятое решение фиксируется одним из способов:

| Тип решения | Где |
|---|---|
| Архитектурное | `docs/adr/NNNN-...md` |
| Бизнесовое (цена, ICP, клиент) | `docs/PRICING.md`, `docs/GTM.md` и т.д. |
| Продуктовое (скоуп фичи) | `docs/EPICS.md` + `specs/07-mvp-acceptance.md` |
| Техдолг / риск | `docs/ARCHITECTURE-REVIEW.md` или приложение В bis в CONCEPT.md |
| UX / дизайн | `docs/UX-REVIEW.md` + Figma (когда появится) |
| Compliance / legal | `docs/COMPLIANCE-MATRIX.md` |

**Правило:** «если решение не в документе — его нет».

## 6. Управление рисками

### 6.1 Reconciliation roadmap как source of truth

58 пунктов в [`REVIEWS-RECONCILIATION.md §5`](./REVIEWS-RECONCILIATION.md). Каждую неделю:
- Отмечать ✅/⏳/❌ по каждому пункту.
- Новые риски — добавляются в тот же документ (не в новые).
- Закрытые риски — НЕ удаляются, помечаются resolved.

### 6.2 Red flags — когда бить тревогу Claude

Обратитесь к Claude немедленно (без weekly session), если:
- **Команда не может начать** эпик из-за blocker'а.
- **Новый клиент / партнёр** появился и требует быстрой оценки.
- **Security-инцидент** любого уровня.
- **Разработчик предлагает** серьёзное архитектурное изменение.
- **Конфликт между** членами команды по техническому вопросу.
- **Изменение законодательства** (особенно 152-ФЗ).

### 6.3 Yellow flags — обсудить на weekly

- Дрифт по срокам какого-то эпика более недели.
- Расход LLM превышает бюджет.
- Проблемы с покупкой компонентов (LLM API, инфра).
- Изменение команды (ушёл разработчик, пришёл новый).

## 7. Шаблоны для типовых ситуаций

### 7.1 «Нужно новое ADR»

Запрос:
> «Я решил [краткое решение] по поводу [проблема]. Обоснование: [причина]. Оформи как ADR.»

Claude:
- Создаёт `docs/adr/NNNN-short-slug.md`.
- Добавляет в `docs/adr/README.md`.
- Обновляет связанные спеки, если нужно.

### 7.2 «Разработчик нашёл проблему»

Запрос:
> «Разработчик пишет: [проблема]. Что посоветовать?»

Claude:
- Анализирует проблему в контексте spec'ов.
- Предлагает 2-3 варианта решения.
- Если это blocker эпика — предлагает workaround на время.

### 7.3 «Клиент просит фичу, которой нет в roadmap»

Запрос:
> «Клиент X попросил [фича]. Оцени: стоит ли делать? Куда добавить?»

Claude:
- Сопоставляет с reconciliation roadmap.
- Оценивает сложность.
- Рекомендует: в MVP / этап 2 / отказать.

### 7.4 «Нужно продать фичу бизнесу»

Запрос:
> «Для [ситуация] нужно [фича]. Напиши коммерческий argument в 3-5 предложениях.»

Claude:
- Даёт текст с упором на ROI.
- Без маркетинговых слов (как договорились в concept).

### 7.5 «Текущая ситуация расходится с документами»

Запрос:
> «В [документ] написано X, а в реальности Y. Что делать?»

Claude:
- Определяет: документ устарел или реальность неправильная.
- Предлагает: обновить документ или исправить реальность.
- Помогает синхронизировать.

## 8. Ограничения Claude

### 8.1 Чего Claude не умеет

- **Не видит состояние live-систем** (production БД, running containers).
- **Не знает, что происходит в мире прямо сейчас** (новые релизы OpenAI, изменения законов).
- **Не помнит предыдущие сессии** без контекста из репо.
- **Не может работать автономно** в фоне между сессиями.
- **Может ошибаться** в технических деталях — нужна проверка.
- **Не заменяет реального senior-инженера** команды.

### 8.2 Чего Claude не должен делать (хотя технически может)

- **Писать production-код.** Только примеры и skeleton'ы.
- **Принимать бизнес-решения.** Только анализ.
- **Оценивать сроки в точных числах.** Только t-shirt sizing (S/M/L/XL).
- **Заменять customer development.** Интервью с клиентами — ваша работа.
- **Обещать что-то клиенту от вашего имени.**

### 8.3 Как проверять Claude

- Сверять с реальным кодом (git).
- Сверять с оригинальными источниками (OpenAI docs, Postgres docs).
- Спрашивать разработчика-человека в команде.
- Если утверждение Claude вызывает сомнения — переформулируйте запрос с просьбой подтвердить или опровергнуть с доказательствами.

## 9. Команды и задачник

### 9.1 Таск-трекер

После найма команды — GitHub Issues / Linear / Jira. До этого:
- `docs/EPICS.md` — список эпиков.
- `docs/REVIEWS-RECONCILIATION.md §5` — roadmap задач.
- `docs/runbooks/` — на каждый инцидент отдельный runbook.

### 9.2 PR-процесс

- Ветки feature/ivan/e4-crud.
- PR template: `.github/PULL_REQUEST_TEMPLATE.md`.
- Review — человек, не Claude.
- После merge — `docs/CHANGELOG.md` обновляется.

## 10. Эскалация и конфликты

### 10.1 Если Claude даёт противоречивые советы в разных сессиях

- Проверить документы — источник правды там.
- Спросить явно: «В прошлой сессии ты сказал X, сейчас Y. Что правильно?»
- Обновить документ, чтобы конфликт не повторялся.

### 10.2 Если разработчик не согласен с Claude

- Разработчик всегда прав в частностях (он видит код).
- Claude прав в архитектурных принципах (он видит систему).
- Разрешение: обновить документ, чтобы отразить реальность + обосновать решение.

### 10.3 Если вы не согласны с Claude

- Claude принимает вашу позицию (вы — руководитель).
- Просите объяснить: «Почему ты считаешь X, а я думаю Y?»
- Может быть, вы правы — это ваш опыт предметной области.

## 13. Ежедневная work life

### 13.1 Без команды (первые недели)

- Claude — ваш единственный «партнёр» по проекту.
- Сессии — по необходимости (обычно 2-3 в неделю для вас).
- Между сессиями — читаете документы, готовите вопросы, взаимодействуете с внешними.

### 13.2 С командой

- Daily standup с командой (без Claude).
- Weekly review с Claude (с протоколом).
- Claude — консультант для team lead'а, а не замена.
- Разработчики сами пишут код, Claude только советует через ревью.

## 14. Коммуникация

### 14.1 Язык

- С Claude — русский (по договорённости).
- В документах — русский.
- В коде — английский (имена, комментарии).

### 14.2 Тон

- Честный. Не стесняйтесь сказать «я не понимаю» — Claude объяснит.
- Конкретный. Абстрактные вопросы дают абстрактные ответы.
- Открытый к критике. Ревью существуют, чтобы находить проблемы.

## 15. Рост ответственности руководителя

На этом проекте (первый) будут **расти**:
- Техническое понимание (от «не разбираюсь» до «могу оценить решение»).
- Бизнес-думание (от «хочу продукт» до «понимаю CAC/LTV/churn»).
- People management (от «найм» до «разрешение конфликтов»).

**Claude помогает в этом росте.** Задавайте вопросы «почему» — не только «что».
