# Recognition: Build vs Buy

**Версия:** 0.1. **Дата:** 2026-04-15. **Источник:** архитектурное ревью J1.

Решение о том, писать ли свой сервис распознавания PDF-спецификаций или использовать готовые решения (Google Document AI, Azure Form Recognizer, Anthropic Claude Vision).

## Текущее решение

**Build** — свой сервис на базе LLM Vision (gpt-4o, gemini-flash). Основа — существующий код `backend/llm_services/services/specification_parser.py`.

## Альтернативы

### Google Document AI
- Специализированное OCR + structured extraction.
- Цена: ~$30 / 1000 страниц = $0.03/страница = ~2.8 ₽/страница.
- Для средней сметы (20 стр.): 56 ₽.
- Плюсы: точность ~95%+, поддержка таблиц, формул, схем.
- Минусы: данные уходят в Google Cloud (США); зависимость от одного вендора; нет control'а над prompt'ами.

### Azure Form Recognizer
- Аналогично Document AI.
- Цена: ~$10 / 1000 страниц = $0.01/страница = ~1 ₽/страница.
- Дешевле Google, немного хуже качество.
- Данные — в Azure (можно выбрать регион).

### Claude Vision (наш текущий подход, но с выделенным промптом под смёточные PDF)
- $0.003/image-tile, средняя страница 4 тайла = $0.012 = 1.1 ₽/страница.
- Плюсы: гибкость промптов, один провайдер для всего.
- Минусы: качество зависит от prompt engineering; требует golden set для валидации.

### Yandex Vision OCR + LLM
- OCR отдельно + LLM поверх.
- Цена: ~₽0.15 за 1 блок = ~3 ₽/страница.
- Плюсы: РФ-контур, 152-ФЗ OK.
- Минусы: двухступенчатый flow, больше инфраструктуры.

## Почему сейчас «Build»

1. **Уже есть прототип** — `SpecificationParser` в ERP. Переписывать с нуля = выкинуть месяцы.
2. **Собственный LLM-стек** уже нужен для агента и matching'а. Ещё один поставщик — усложнение.
3. **Гибкость для разных отраслей** — если расширяемся на окна/двери, промпты легко адаптируются. У готовых сервисов — только «общий» режим.
4. **152-ФЗ контроль** — мы сами выбираем, какой LLM использовать. У Google/Azure выбор ограничен.

## Триггер пересмотра

После MVP — провести формальный бенчмарк на golden set из 10 смет:
- precision recognition (доля корректно извлечённых позиций);
- recall recognition (доля найденных позиций);
- latency;
- стоимость.

Методика:
1. Размечаем эталон: на каждой смете 10 специалист вручную указывает точный список позиций.
2. Прогоняем через все 4 варианта (наш + Google + Azure + Yandex).
3. Сравниваем с эталоном: precision/recall/F1.
4. Добавляем стоимость и latency.

Если один из готовых сервисов даёт:
- precision > наш +5%;
- и стоимость не выше на 50%;

→ Переходим на него (по крайней мере для default-режима; fallback на свой остаётся).
