# Модель LLM-затрат ISMeta

**Версия:** 0.1. **Дата:** 2026-04-15. **Источник:** архитектурное ревью F1.

Реальная оценка расходов на LLM в разрезе задач и размеров смет. Основа для бюджетирования, ценообразования коробочного продукта и выбора моделей.

Цены актуальны на дату документа. Обновлять ежеквартально.

## 1. Единицы учёта

Все расчёты — в USD (фактические ставки провайдеров) и ₽ (по курсу ЦБР на дату расчёта, 1 USD ≈ 92.5 ₽).

1 LLM-вызов = 1 запрос к провайдеру (включая tool-вызовы).
1 задача = один бизнес-флоу (например, «подбор работ для сметы», включает N LLM-вызовов).

## 3. Типовые задачи и их стоимость

### 3.1 Распознавание PDF-спецификации (recognition)

Вызывается при загрузке спецификации.

Параметры:
- страниц: 5-50 (средняя смета 2000 строк ≈ 20 страниц PDF);
- input: 1 страница PNG в base64 + system prompt ≈ 1500 токенов (с учётом картинки);
- output: структурированный JSON из 100 строк ≈ 3500 токенов.

**Расчёт на среднюю смету (20 страниц):**
- input: 20 × 1500 = 30 000 токенов;
- output: 20 × 3500 = 70 000 токенов (разбивается по 5 страниц в батч);
- gpt-4o: 30K × $2.50/M + 70K × $10/M = $0.075 + $0.70 = **$0.775 = 72 ₽**.

**На малую смету (5 страниц):** $0.20 = **18 ₽**.
**На крупную смету (50 страниц):** $1.90 = **176 ₽**.

Оптимизация: gpt-4o-mini для первого прохода, gpt-4o только для повторного валидирования низко-confidence страниц. Экономия — до 70%.

### 3.2 Подбор работ Tier 6 (LLM semantic match)

Вызывается только для позиций, где Tier 0-5 не дали уверенного результата. На типичной смете — 10-20% позиций.

Параметры:
- батч: 5 позиций + 25 кандидатов (топ-5 каждой) = ~2500 токенов input;
- output: JSON match'ей, ~500 токенов;
- смета 2000 строк, 15% unmatched = 300 позиций = 60 батчей.

**Расчёт:**
- input: 60 × 2500 = 150 000 токенов;
- output: 60 × 500 = 30 000 токенов;
- gpt-4o: $0.375 + $0.30 = **$0.675 = 62 ₽**.

**На малую смету (100 строк, 15% = 3 батча):** $0.034 = **3.1 ₽**.
**На крупную смету (10 000 строк, 15% = 300 батчей):** $3.40 = **315 ₽**.

Оптимизация: gpt-4o-mini достаточно для большинства случаев. Экономия — 95%.
- gpt-4o-mini средняя смета: **$0.045 = 4.1 ₽**.

### 3.3 Подбор работ Tier 7 (LLM + web search)

Последний шанс для остатка позиций. ~5% после Tier 6.

Параметры:
- per item: 1 запрос с web grounding, ~3000 токенов input, ~1000 output;
- смета 2000 строк, ~5% = 100 items = 100 запросов.

**Расчёт:**
- gemini-2.5-flash (с web-grounding): 100 × (3000 × $0.30/M + 1000 × $2.50/M) = 100 × ($0.0009 + $0.0025) = $0.34 = **31 ₽**.
- Плюс grounding cost ≈ $0.002/запрос = $0.20 = **18 ₽**.
- **Итого Tier 7 средняя смета: 49 ₽**.

### 3.4 LLM-агент (чат сметчика)

Активность: 5-20 сообщений на смету в среднем (если сметчик пользуется чатом).

Параметры:
- system prompt + история: ~800 токенов;
- tool call + tool result: ~500 токенов;
- текстовый ответ: ~300 токенов;
- одно сообщение = 2 LLM-вызова (первичный + ответ после tool).

**Расчёт на 10 сообщений:**
- суммарный input: 10 × 2 × 1300 = 26 000 токенов;
- суммарный output: 10 × 2 × 500 = 10 000 токенов;
- gpt-4o: $0.065 + $0.10 = **$0.165 = 15 ₽**.

На 20 сообщений: **30 ₽**.

**Таргет 50 ₽ на среднюю смету** — достижим только при агрессивной оптимизации:

- Tier 7: gemini с web search (уже оптимально);
- агент: gpt-4o-mini для простых tool-вызовов, gpt-4o для рассуждений.

**Честный таргет:** 100-150 ₽ на среднюю смету без оптимизации, 35-50 ₽ с.

## 5. Budget enforcement

### 5.1 На уровне Workspace

В `Workspace.settings.llm_budget`:
```json
{
  "monthly_budget_rub": 10000,
  "daily_budget_rub": 500,
  "alert_at_percent": 80,
  "block_at_percent": 100,
  "model_downgrade_at_percent": 70
}
```

### 5.2 Автоматика

Celery-beat task `check_llm_budget` каждые 5 минут:
1. Считает `SUM(cost_usd)` из `LLMUsage` за текущий месяц.
2. Сравнивает с `monthly_budget_rub`.
3. При 70% — переключает задачи на `model_downgrade_map` (gpt-4o → gpt-4o-mini).
4. При 80% — отправляет алерт.
5. При 100% — блокирует новые LLM-вызовы, пользователь видит сообщение «LLM-бюджет исчерпан, обратитесь к администратору».

### 5.3 Per-task override

Админ может задать budget per task type (например, recognition не блокировать даже при 100%, потому что это критический flow).

## 6. Биллинг для коробочных клиентов

### 6.1 Варианты

**Вариант A: клиент платит своим ключом.**
- Клиент получает собственный OpenAI/Gemini key.
- ISMeta прокидывает запросы через его ключ.
- Мы видим usage в нашем LLMUsage, но платит клиент напрямую провайдеру.
- Плюс: простота, прозрачность.
- Минус: клиенту надо оформлять аккаунт у провайдера.

**Вариант B: сквозной биллинг.**
- Клиент платит нам, мы покупаем токены.
- Мы добавляем свою маржу (5-15%).
- Плюс: удобство клиенту.
- Минус: юридически сложнее (мы становимся reseller'ом), cashflow-риски.

**Вариант C: flat tariff.**
- Клиент платит фиксированную цену в месяц за N смет.
- Мы управляем моделями и оптимизацией.
- Плюс: предсказуемость для клиента.
- Минус: наш риск при неоптимальной модели.

**Рекомендация для MVP:** вариант A. Для SaaS — вариант C.

### 6.2 Структура flat tariff (черновик для обсуждения)

| Тариф | Цена/мес | Включено | Средняя cost/смета |
|---|---|---|---|
| Старт | 30 000 ₽ | до 20 смет | 1500 ₽/смета — покрывает 50 ₽ LLM × 10 запас |
| Бизнес | 100 000 ₽ | до 100 смет | 1000 ₽/смета |
| Enterprise | 300 000 ₽ | до 500 смет | 600 ₽/смета |

Реальные цены — определяет продукт после MVP.

## 7. Что отслеживать

- `llm_cost_rub_per_estimate` (p50, p95, max);
- `llm_cost_rub_per_workspace` (помесячно);
- `llm_cost_rub_per_task_type`;
- доля смет, превысивших бюджет;
- доля задач, переключенных на downgrade-модель.

## 8. Ревизия

Модель пересматривается:
- ежеквартально (актуализация цен провайдеров);
- при смене default-модели (например, появилась более дешёвая);
- при первой коробочной продаже (реальные данные клиента корректируют прикидки).
