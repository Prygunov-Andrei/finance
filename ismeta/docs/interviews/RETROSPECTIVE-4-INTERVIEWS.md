## 3. Автоподбор работ через LLM — переосмысление

| Кто | Мнение |
|---|---|
| Даша | Не упомянула как боль |
| Сергей | Расценки в голове, не нужен |
| Катя | **Скептична:** «расслаблюсь → пропущу коммутатор за 800 тысяч» |
| Оля | **«ИИ как ПРОВЕРЯЮЩИЙ»:** «загрузить готовую смету → ИИ указывает ошибки» |

### Решение: два режима ИИ

1. **Режим «помощник/проверяющий»** (внутренний продукт для сметчиков):
   - Не автозаполнение, а подсказки с вариантами.
   - Confidence score на каждой позиции.
   - Обязательный ручной confirm на позициях > порога.
   - «Проверь мою смету» — загрузить готовую → ИИ указывает подозрительное.

2. **Режим «автомат»** (публичная часть сайта для клиентов):
   - Полная автоматика: клиент загружает спецификацию → получает КП.
   - Ожидаемая точность ниже — но клиент не профессионал, ему достаточно.
   - Дисклеймер: «ориентировочная оценка, точная — после проверки сметчиком».

## 8. Стратегическое решение: полукоробка

**Принято по предложению Андрея.**

Формула: **продукт под процесс Август Климат, с архитектурой для лёгкой адаптации.**

Что это значит для кода:
- UI и workflow = конкретный процесс Августа (PDF → смета → поставщики → ТКП → передача в ERP).
- Конфиги (коэффициенты, наценки, форматы экспорта) = в `WorkspaceSettings`, не в коде.
- Прайс монтажа = отдельная сущность, загружаемая per workspace.
- Поставщики = per workspace.
- Формат ТКП = template per workspace/customer.

Нужен **ADR-0022** для фиксации.
