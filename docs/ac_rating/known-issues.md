# Known issues — рейтинг кондиционеров

Не-блокирующие замечания, выявленные аудитом математики scoring engine. Максим закрыл 20 из 23 пунктов в коммите `e2de2de` (ветка `2026-03-25-xuef`); ниже — оставшиеся 3, которые мы осознанно принимаем и фиксим позже.

## 1. Fan speeds gap — пропуски ступеней не учитываются при скоринге

**Статус:** accepted
**Почему не блокирует:** на типичных моделях кондиционеров число ступеней вентилятора берётся либо из заявленных в каталоге, либо из лабораторных измерений. На текущем датасете «гэпов» (например, заявлено 5 ступеней, но физически 1, 2, 5) в сырых данных нет, и оценка через `NumericScorer` (min=низко, max=высоко) даёт корректное направление шкалы. На контрольных 5 моделях, по которым Максим сверял с прошлым прод-индексом, расхождение < 0.1.

**Локация:**
- `backend/ac_scoring/scorers/numeric.py` — основной алгоритм оценки числовых критериев.
- `backend/ac_scoring/scorers/categorical.py` (`UNIVERSAL_SCALE`) — фолбэк для текстовых ответов вида «5 ступеней / 3 ступени». Здесь и теряется информация о пропусках: «3» и «3 с пропуском 5-й» оцениваются одинаково.

**Возможное будущее решение:** ввести отдельный `value_type='fan_speeds'` или расширить formula_json схемой ступеней (массив активных ступеней), и завести `FanSpeedsScorer`, который штрафует за gap. Имеет смысл сделать вместе с фазой Ф8B (UI методики), когда появится явный input «активные ступени».

## 2. Decimal vs Float — балл индекса хранится как float

**Статус:** accepted
**Почему не блокирует:** итоговый индекс — число в диапазоне 0..100 с двумя знаками после запятой. Точность IEEE 754 double заведомо превышает требуемую (порядка 1e-12 на шаге, при том что `round(..., 2)` отрезает на 1e-2). В ERP `Decimal` используется для денег и для весов в накладных, где разница в копейке/грамме критична; для рейтингового балла это излишество.

**Локация:**
- `backend/ac_methodology/models.py:138-150` — `weight`, `min_value`, `median_value`, `max_value` все `FloatField`.
- `backend/ac_catalog/models.py:38-40` — `ACModel.nominal_capacity` `FloatField`.
- `backend/ac_catalog/models.py:155-156` — `ModelRawValue.numeric_value` `FloatField`.
- `backend/ac_scoring/models.py:50-53` — `CalculationResult.normalized_score`, `weighted_score` `FloatField`.
- Все скореры в `backend/ac_scoring/scorers/` оперируют `float` (`raw_value` приходит строкой → `float()`); это согласовано с типом колонок выше.

**Решение и обоснование:** *оставляем `float` для всего scoring-pipeline.* Если в будущем понадобится связать индекс с финансовыми метриками (например, гарантированный балл в договоре), пересмотрим — `Decimal` ввести можно изолировано в `CalculationResult.weighted_score`, не ломая остальное.

## 3. Help_text — нет пояснений у части полей методики

**Статус:** accepted (UX-долг)
**Почему не блокирует:** на MVP методику редактирует только Андрей через Django admin (фаза Ф4). Поля семантически прозрачны опытному пользователю; справка пока в `ac-rating/plan.md` и `brief-designer.md`.

**Локация (поля без `help_text`):**
- `backend/ac_methodology/models.py` `Criterion`:
  - `name_ru/en/de/pt` (74-77), `description_ru/en/de/pt` (79-82), `unit` (84), `value_type` (91-93), `is_active` (95).
- `backend/ac_methodology/models.py` `MethodologyCriterion`:
  - `scoring_type` (133-135), `min_value/median_value/max_value` (141-143), `is_required_*` (164-166), `use_in_*` (168-170), `region_scope` (172-175), `is_public` (176), `display_order` (177), `is_active` (178).
- `backend/ac_methodology/models.py` `MethodologyVersion`:
  - `tab_description_*` (13-24) — длинные текстовые поля без подсказки, что именно туда писать.

**Возможное будущее решение:** в Ф8B (UI методики) переписываем форму с inline-help-блоками; одновременно проходим по моделям и добавляем `help_text` для admin-fallback. Тексты подсказок подберём вместе с Андреем при дизайн-сессии методики.
